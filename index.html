ตี 3 คลัสเตอร์ล่ม — แล้วสมชายก็รู้ว่า "production" ของตัวเองไม่เคยเป็น production จริง

🏗️ บทที่ 2

Control Plane กับ Node ต่างกันยังไง แล้ว request หนึ่งอันวิ่งผ่านใครบ้าง

คลัสเตอร์เดี่ยวของสมชายที่ล่มตอนตีสามในบทที่แล้ว มันคือ VM เดียวที่ยัดทุกอย่างไว้ด้วยกัน — พอ process ไหน process หนึ่งค้าง ทุกอย่างค้างตาม บทนี้เราจะแกะดูว่าจริง ๆ แล้ว Kubernetes มี component อะไรบ้าง แต่ละตัวทำหน้าที่อะไร และทำไมการแยกมันออกจากกันถึงเป็นจุดเริ่มต้นของความ HA

🧑‍💻
สมชายผมงงตรงนี้ครับพี่ ก. ตอนคลัสเตอร์ผมล่ม ผม SSH เข้าไปดู เจอ process ชื่อ kube-apiserver, etcd, scheduler เต็มไปหมด นี่มันคนละโปรแกรมกันเลยเหรอ ไม่ใช่ "Kubernetes" ตัวเดียว?
🧑‍🏫
นาย ก.ใช่ครับ "Kubernetes" ที่เราพูดถึงกันมันคือชื่อรวมของ component หลายตัวที่ทำงานร่วมกัน ตอนคุณรัน VM เดียว ทุก component เลยไปนั่งอยู่บนเครื่องเดียวกันหมด พอ disk เต็มหรือ CPU พีค ทุกตัวพังพร้อมกันเป็นโดมิโน

กล่าวสั้น ๆ คือ Kubernetes แบ่งงานเป็นสองฝั่งชัดเจน — control plane ที่ตัดสินใจว่าอะไรควรเกิดขึ้น กับ node component ที่ทำให้มันเกิดขึ้นจริงบนเครื่อง

ทีนี้พอเห็นภาพ flow แล้ว มาดูกันทีละตัวว่าใครอยู่ฝั่งไหน และถ้าตัวนั้นตายไปเดี่ยว ๆ จะเกิดอะไรขึ้นกับคลัสเตอร์

Componentอยู่ฝั่งไหนทำหน้าที่อะไรถ้าตัวนี้ล่ม เกิดอะไรขึ้น
kube-apiserverControl planeหน้าประตู REST API ของทั้งคลัสเตอร์ ทุกคำสั่งผ่านตรงนี้หมดเข้าคลัสเตอร์ไม่ได้เลย kubectl ค้างหรือ error ทันที
etcdControl planeเก็บ state ทั้งหมดของคลัสเตอร์แบบ key-value ที่ consistentapiserver อ่าน/เขียน state ไม่ได้ ถือว่าคลัสเตอร์ตายทั้งระบบ
kube-schedulerControl planeเลือกโหนดที่เหมาะสมให้ Pod ที่ยังไม่ถูก assignPod เดิมยังรันอยู่ปกติ แต่ Pod ใหม่ค้างสถานะ Pending ตลอด

ลองดูของจริงกัน — ถ้าคลัสเตอร์ตั้งด้วย kubeadm แบบ stacked (ซึ่งเป็นสิ่งที่เราจะทำในบทที่ 5) control plane component เหล่านี้จะถูกรันเป็น static pod อยู่ใน namespace kube-system ลองสั่งดูได้เลย

terminal
# ดู pod ทั้งหมดใน control plane namespace
kubectl get pods -n kube-system -o wide

อธิบายจาก output

  1. kube-apiserver-<nodename> — static pod ของ apiserver หนึ่งตัวต่อหนึ่ง control-plane node
  2. etcd-<nodename> — จะเห็นก็ต่อเมื่อใช้ stacked etcd (รายละเอียดในบทที่ 3) ถ้าใช้ external etcd จะไม่มี pod นี้ในนี้
  3. kube-scheduler-<nodename> และ kube-controller-manager-<nodename> — รันแบบ leader-election ถ้ามีหลาย control-plane node จะมีหลาย pod แต่มีตัวเดียวที่ "active" จริง
  4. kube-proxy-xxxxx — เป็น DaemonSet รันบนทุกโหนดรวมทั้ง worker
  5. คอลัมน์ -o wide จะโชว์ NODE ให้เห็นว่า pod แต่ละตัวไปนั่งอยู่บนเครื่องไหน — ใช้เช็คได้เลยว่า control-plane node แต่ละตัว "ครบชุด" จริงไหม

💡 สองตัวที่ "optional"

kube-proxy และ cloud-controller-manager เป็น component ที่ทางการเรียกว่า optional — kube-proxy เพราะบาง CNI (เช่นบาง mode ของ Cilium) แทนที่มันด้วย eBPF ได้ ส่วน cloud-controller-manager มีไว้ผูกกับ cloud provider (สร้าง LoadBalancer, จัดการ node ที่หายไปจาก cloud ฯลฯ) ซึ่งบน bare-metal ล้วน ๆ แบบที่เราจะทำในซีรีส์นี้ไม่ต้องมีเลย

รู้จัก component ครบแล้ว คำถามต่อไปคือ — เอา component พวกนี้ไปวางบนกี่เครื่องถึงจะเรียกว่า HA จริง ๆ ไปดูกันในบทที่ 3

🧭 บทที่ 3

Stacked etcd หรือ External etcd — ต้องใช้กี่เครื่องกันแน่ถึงจะ HA

รู้ component แล้วอย่างเดียวยังไม่พอ ต้องรู้ด้วยว่าจะ "กระจาย" component พวกนี้ไปวางบนกี่เครื่องยังไงถึงจะรอดเมื่อมีเครื่องใดเครื่องหนึ่งตาย บทนี้เทียบให้เห็นสอง topology ที่ kubeadm รองรับ พร้อมเรื่อง CIDR ที่ต้องวางแผนตั้งแต่ต้น

🙋
นาย ข.พี่ครับ ถ้าจะทำ HA ผมมองว่าก็แค่เพิ่ม control-plane node เป็น 2 เครื่องก็พอแล้วใช่มั้ยครับ เผื่อเครื่องนึงตาย อีกเครื่องรับแทน?
🧑‍🏫
นาย ก.ผิดครับ นั่นคือกับดักที่คนตั้ง HA cluster ครั้งแรกชอบพลาด — etcd ใช้ Raft consensus ต้องมี "เสียงข้างมาก" (quorum) ถึงจะเขียนข้อมูลได้ ถ้ามีแค่ 2 โหนด พอโหนดนึงตาย อีกโหนดเหลือ 1 ใน 2 เสียง ไม่ถึงครึ่ง คลัสเตอร์ก็เขียนอะไรไม่ได้เหมือนกัน ต้องขั้นต่ำ 3 เครื่องเสมอ

กล่าวสั้น ๆ คือ HA ของ Kubernetes ผูกกับ quorum ของ etcd โดยตรง — ขั้นต่ำต้องมี etcd 3 ตัว (เลขคี่ เพื่อให้มี majority ชัดเจน) ไม่ว่าจะเลือกวางมันไว้บนเครื่องเดียวกับ control plane (stacked) หรือแยกเครื่องต่างหาก (external)

kubeadm รองรับสอง topology หลัก — เทียบให้เห็นชัด ๆ ในตารางนี้

Fault isolationต่ำกว่า — ถ้าเครื่อง control-plane node ตาย เสีย control plane + etcd พร้อมกันในโหนดเดียวสูงกว่า — apiserver แต่ละตัวคุยกับ etcd cluster แยกต่างหาก ปัญหาฝั่งใดฝั่งหนึ่งไม่ลากอีกฝั่งตาย

ความซับซ้อนในการดูแลต่ำกว่า — kubeadm สร้าง etcd static pod ให้อัตโนมัติตอน kubeadm init / kubeadm join --control-planeสูงกว่า — ต้องตั้ง etcd cluster เองแยกก่อน แล้วป้อน endpoint ให้ apiserver ตอน init
เหมาะกับทีมเล็ก-กลาง, งบจำกัด, อยากเริ่ม HA ให้ไวที่สุดองค์กรใหญ่ที่ต้องการแยก failure domain ของ data layer ออกจาก control layer ชัดเจน

เอกสารทางการของ kubeadm พูดไว้ตรง ๆ ว่า "kubeadm bootstraps the etcd cluster statically" — คือไม่ว่าจะเลือกแบบไหน kubeadm ก็จัดการ etcd cluster ให้แบบ static ไม่ต้องมี etcd operator แยกต่างหาก สำหรับซีรีส์นี้เราจะใช้ stacked etcd เพราะจำนวนเครื่องคุ้มกว่าและพอสำหรับ production ระดับกลางได้สบาย

ทีนี้ไม่ว่าจะมี control-plane node กี่ตัว worker ก็ยังต้องรู้จัก apiserver แค่ "จุดเดียว" อยู่ดี — นี่คือที่มาของ --control-plane-endpoint ซึ่งควรชี้ไปที่ load balancer (หรืออย่างน้อยที่สุดคือ DNS name/VIP หนึ่งจุด) ที่กระจาย traffic ไปยัง apiserver ทุกตัว ไม่ใช่ชี้ไป control-plane node ตัวใดตัวหนึ่งตรง ๆ

/etc/hosts (ทุกเครื่องในคลัสเตอร์)
# ชี้ชื่อ endpoint กลางไปที่ VIP ของ load balancer
192.168.0.100  cluster-endpoint

อธิบายจาก config

  1. 192.168.0.100 — IP ของ load balancer (VIP) ที่ยืนหน้า apiserver ทุกตัว ไม่ใช่ IP ของ control-plane node เครื่องใดเครื่องหนึ่ง
  2. cluster-endpoint — ชื่อที่จะถูกส่งเข้า kubeadm init --control-plane-endpoint=cluster-endpoint ในบทที่ 5 — ทุกเครื่อง (control-plane และ worker) ต้อง resolve ชื่อนี้ไปที่ VIP เดียวกันได้
  3. เวลามี control-plane node ตายไปตัวหนึ่ง load balancer จะเลิกส่ง traffic ไปตัวนั้น worker ก็ยังต่อ apiserver ที่เหลือได้โดยไม่ต้องแก้ config อะไรเลย

💡 /etc/hosts พอสำหรับ lab แต่ไม่พอสำหรับ production

ตัวอย่างข้างบนคือวิธีที่ง่ายที่สุดสำหรับทดสอบในวง lab เท่านั้น งาน production จริงควรใช้ load balancer ที่ทำ health check จริง เช่น HAProxy + keepalived (ทำ VIP แบบ active-passive) หรือ external load balancer ของ network ที่มีอยู่แล้ว ไม่งั้นถ้า control-plane node ที่ IP ตรงกับ /etc/hosts ตายไป ทุกอย่างจะพังตามเพราะไม่มีใคร failover ให้

อีกเรื่องที่ต้องวางแผนคู่กับ topology คือ CIDR — pod-network-cidr ที่จะป้อนตอน kubeadm init ต้องไม่ชนกับ network เดิมที่มีอยู่แล้ว (เช่น network ของ VM เอง หรือ office LAN) และต้องตรงกับค่า default ของ CNI ที่จะติดตั้ง — เดี๋ยวจะเห็นว่า Calico ที่เราจะใช้ในบทที่ 5 มีค่า default เป็น 192.168.0.0/16 พอดี ถ้าเลข CIDR ไม่ตรงกัน Pod จะขอ IP ไม่ได้เลยตั้งแต่ต้น

วางแผน topology และ IP เสร็จแล้ว ต่อไปคือเอามือลงไปเตรียมตัวเครื่องจริง ๆ ก่อนจะรัน kubeadm ได้ — ไปดูกันในบทที่ 4

ก่อนจะรัน kubeadm init ได้แม้แต่ครั้งเดียว มีการบ้านที่ต้องทำบนทุก VM ก่อน — พลาดข้อไหนข้อหนึ่ง kubeadm จะฟ้อง preflight error ทันที บทนี้ไล่ทีละขั้นตอนแบบที่ทำได้จริง

สมชายผมลอง kubeadm init เลยครับ มัน error ว่า swap enabled แล้วก็ยังบ่นเรื่อง bridge-nf อะไรสักอย่างอีก งงเลย ทำไมต้องเซ็ตอะไรพวกนี้ก่อนด้วย
🧑‍🏫
นาย ก.เพราะ kubelet ไม่รองรับ swap ตั้งแต่ต้น มัน scheduling decision จะเพี้ยนถ้า process ไป swap แทนที่จะโดน OOM kill ตามที่ควร ส่วน bridge-nf คือให้ iptables มองเห็น traffic ที่ผ่าน bridge network ได้ ซึ่ง kube-proxy ต้องพึ่งอันนี้ทำงาน ทั้งหมดนี้ต้องเซ็ตให้ครบก่อน ไม่งั้น kubeadm preflight check จะบล็อกไม่ให้ init เลย

กล่าวสั้น ๆ คือ การเตรียม VM คือการปิด swap, เปิด kernel module ที่ networking ของ Kubernetes ต้องใช้, เซ็ต sysctl ให้ traffic ผ่าน bridge ได้ถูกต้อง, ติดตั้ง container runtime ให้ cgroup driver ตรงกับ kubelet และเปิด firewall port ที่จำเป็น

เริ่มจากปิด swap ก่อน — รันบนทุก node ทั้ง control-plane และ worker

ทุก node
# ปิด swap ทันที
sudo swapoff -a

# ปิดถาวร — comment บรรทัด swap ออกจาก /etc/fstab
sudo sed -i '/\sswap\s/s/^/#/' /etc/fstab

อธิบายจาก command

  1. swapoff -a — ปิด swap partition/file ทั้งหมดที่ active อยู่ตอนนี้ทันที มีผลจนกว่าจะ reboot
  2. บรรทัดที่สอง — comment บรรทัดที่มีคำว่า swap ใน /etc/fstab ออก เพื่อไม่ให้ swap ถูก mount กลับมาอัตโนมัติตอน reboot เครื่อง
  3. เช็คผลได้ด้วย free -h — คอลัมน์ Swap ต้องเป็น 0 ทั้งหมด

ต่อมาคือ kernel module สองตัวที่ networking ของ Kubernetes ต้องใช้ — overlay สำหรับ container runtime และ br_netfilter สำหรับให้ bridge traffic ผ่าน iptables ได้

ทุก node
# โหลด module อัตโนมัติทุกครั้งที่บูต
cat <<EOF | sudo tee /etc/modules-load.d/k8s.conf
overlay
br_netfilter
EOF

# โหลดทันทีโดยไม่ต้อง reboot
sudo modprobe overlay
sudo modprobe br_netfilter

อธิบายจาก command

  1. /etc/modules-load.d/k8s.conf — ไฟล์ config บอก systemd ว่าต้องโหลด module ไหนบ้างตอนบูตเครื่อง เขียนชื่อ module บรรทัดละตัว
  2. overlay — filesystem driver ที่ container runtime อย่าง containerd ใช้ทำ layered image
  3. br_netfilter — ทำให้ traffic ที่วิ่งผ่าน Linux bridge ถูกส่งผ่าน iptables chain ด้วย ซึ่งจำเป็นสำหรับ kube-proxy และ CNI plugin ส่วนใหญ่
  4. modprobe สองบรรทัดสุดท้าย — โหลด module เข้า kernel ทันทีโดยไม่ต้องรอ reboot เช็คได้ด้วย lsmod | grep br_netfilter

ต่อด้วย sysctl ที่ผูกกับ module ข้างบนโดยตรง เพื่อเปิดให้ bridge traffic ผ่าน iptables จริง ๆ และเปิด IP forwarding ให้ node ทำหน้าที่ router ระหว่าง pod ได้

ทุก node
cat <<EOF | sudo tee /etc/sysctl.d/k8s.conf
net.bridge.bridge-nf-call-iptables  = 1
net.bridge.bridge-nf-call-ip6tables = 1
net.ipv4.ip_forward                 = 1
EOF

# apply โดยไม่ต้อง reboot
sudo sysctl --system

อธิบายจาก config

  1. net.bridge.bridge-nf-call-iptables / ip6tables — บังคับให้ traffic ที่ผ่าน bridge ถูกประเมินโดย iptables/ip6tables rule ด้วย ไม่งั้น kube-proxy เขียน rule ไปก็ไม่มีผลจริง
  2. net.ipv4.ip_forward = 1 — เปิดให้เครื่องนี้ forward packet ข้าม interface ได้ ซึ่งจำเป็นเพราะ pod บนโหนดหนึ่งต้องคุยกับ pod อีกโหนดผ่าน node เป็นทางผ่าน
  3. sysctl --system — โหลด config ทุกไฟล์ใน /etc/sysctl.d/ เข้าไปใช้จริงทันที ไม่ต้อง reboot
  4. เช็คผลด้วย sysctl net.ipv4.ip_forward — ต้องได้ค่ากลับมาเป็น net.ipv4.ip_forward = 1

ถัดมาคือ container runtime — เราจะใช้ containerd เพราะเป็นตัวที่ community ใช้กันมากที่สุดกับ kubeadm ขั้นตอนคือ generate config default มาก่อน แล้วค่อยเปิด SystemdCgroup

/etc/containerd/config.toml
# generate default config ก่อน (ทับของเดิมถ้ามี)
containerd config default | sudo tee /etc/containerd/config.toml

# containerd 1.x — path เดิม
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.runc.options]
  SystemdCgroup = true

# containerd 2.x — path เปลี่ยน! (grpc.v1.cri → cri.v1.runtime)
[plugins.'io.containerd.cri.v1.runtime'.containerd.runtimes.runc.options]
  SystemdCgroup = true

sudo systemctl restart containerd

อธิบายจาก config

  1. containerd config default — generate ไฟล์ config เริ่มต้นที่ถูกต้องตามเวอร์ชันที่ติดตั้งจริง เช็คเวอร์ชันก่อนด้วย containerd --version
  2. path ของ [plugins...runc.options] — ต้องเลือกให้ตรงกับเวอร์ชัน containerd ที่ลงจริง ถ้าใช้ path ผิดเวอร์ชัน block นี้จะไม่มีผลอะไรเลยเงียบ ๆ (silent no-op)
  3. SystemdCgroup = true — บอก containerd ให้ใช้ systemd เป็น cgroup driver ต้องตรงกับ cgroup driver ของ kubelet (ค่า default ของ kubelet เวอร์ชันใหม่คือ systemd อยู่แล้ว) ถ้าไม่ตรงกัน node จะไม่เข้า Ready
  4. systemctl restart containerd — apply config ใหม่ ต้องรันทุกครั้งหลังแก้ config.toml

⚠️ containerd 1.x กับ 2.x ใช้ path ต่างกัน

นี่คือสาเหตุอันดับต้น ๆ ที่คลัสเตอร์เพิ่งตั้งใหม่แล้ว node ค้าง NotReady ทั้งที่ทำตาม guide เป๊ะ — เอา config แบบ containerd 1.x ไปใส่ในเครื่องที่ลง containerd 2.x (หรือกลับกัน) แล้ว path ไม่ match block นั้นจะไม่ error ให้เห็น แค่ไม่มีผลอะไรเลย เช็ค containerd --version ก่อนแก้ config ทุกครั้ง

อธิบายจาก command

  1. nc 127.0.0.1 6443 -zv -w 2 — ทดสอบว่า port 6443 เปิดรับ connection อยู่ไหม (-z คือแค่เช็ค ไม่ส่งข้อมูล, -w 2 คือ timeout 2 วินาที) ใช้ debug ตอน worker join แล้วต่อ apiserver ไม่ติด
  2. ip link — โชว์ MAC address ของทุก network interface เอาไปเทียบข้ามเครื่อง ถ้า VM ถูก clone มาจาก template เดียวกันแบบไม่เปลี่ยน MAC อาจชนกันได้ ซึ่ง kubeadm preflight จะ error
  3. product_uuid — UUID เฉพาะเครื่องที่ kubeadm ใช้ระบุตัวตนของแต่ละโหนด ถ้า VM clone มาแบบเดียวกันแล้ว UUID ซ้ำกัน kubeadm จะปฏิเสธไม่ให้ join

VM ทุกเครื่องพร้อมแล้ว — swap ปิด, module โหลด, sysctl เซ็ต, containerd รันด้วย cgroup driver ที่ถูกต้อง, firewall เปิด port ครบ ถึงเวลาลง kubeadm จริง ๆ และรัน init ครั้งแรกในบทที่ 5

🚀 บทที่ 5

Bootstrap คลัสเตอร์จริง — kubeadm init แบบ HA จนถึง CNI พร้อมใช้งาน

VM พร้อมหมดแล้วจากบทที่แล้ว บทนี้คือขั้นตอนตัวจริง — ติดตั้ง kubeadm, รัน init แบบ HA, join worker กับ control-plane node เพิ่ม แล้วปิดท้ายด้วยการลง Calico ให้ Pod คุยกันได้

ติดตั้ง kubelet, kubeadm, kubectl บนทุก node (ทั้ง control-plane และ worker) ด้วย repo ปัจจุบัน

ทุก node (Debian/Ubuntu)
sudo apt-get update
sudo apt-get install -y apt-transport-https ca-certificates curl gpg

# ดึง GPG key จาก repo ปัจจุบัน (ไม่ใช่ apt-key add แบบเก่า)
curl -fsSL https://pkgs.k8s.io/core:/stable:/v1.31/deb/Release.key | \
  sudo gpg --dearmor -o /etc/apt/keyrings/kubernetes-apt-keyring.gpg

# เพิ่ม repo ของ minor version ที่ต้องการ
echo 'deb [signed-by=/etc/apt/keyrings/kubernetes-apt-keyring.gpg] https://pkgs.k8s.io/core:/stable:/v1.31/deb/ /' | \
  sudo tee /etc/apt/sources.list.d/kubernetes.list

sudo apt-get update
sudo apt-get install -y kubelet kubeadm kubectl

# กัน apt upgrade เผลอเปลี่ยนเวอร์ชันแบบไม่ตั้งใจ
sudo apt-mark hold kubelet kubeadm kubectl

อธิบายจาก command

  1. pkgs.k8s.io/core:/stable:/v1.31/deb/ — repo แยกตาม minor version ชัดเจน เลข v1.31 ต้องเปลี่ยนตามเวอร์ชันเป้าหมายของคลัสเตอร์คุณ
  2. GPG key ผ่าน gpg --dearmor — วิธีปัจจุบันที่แทนที่ apt-key add ที่ deprecated ไปแล้วเช่นกัน
  3. apt-mark hold — ล็อกเวอร์ชันไว้ ป้องกัน apt upgrade ปกติดันเวอร์ชัน kubelet ให้สูงกว่า apiserver โดยไม่ตั้งใจ (กติกา version skew คือ kubelet ตามหลัง apiserver ได้ไม่เกิน 1 minor version และห้ามนำหน้าเด็ดขาด)
  4. ทำ 4 บรรทัดนี้ให้ครบบนทุก node ไม่ใช่แค่ control-plane — worker ก็ต้องมี kubelet + kubeadm เหมือนกัน (kubectl จริง ๆ จำเป็นเฉพาะเครื่องที่จะสั่งคลัสเตอร์)

💡 เช็ค version ล่าสุดก่อนใช้งานจริง

v1.31 ในตัวอย่างนี้คือ minor version ที่เสถียรและใช้กันแพร่หลายตอนเขียนบทความนี้ ให้เช็ค release ล่าสุดที่ kubernetes.io/releases ก่อนเลือกใช้จริง แล้วแทนที่เลข v1.31 ทุกจุดในคำสั่งข้างบนให้ตรงกัน

ติดตั้งครบทุกเครื่องแล้ว มาเริ่ม init คลัสเตอร์ที่ control-plane node ตัวแรก — ใช้ฟอร์ม HA ที่มี --control-plane-endpoint และ --upload-certs

control-plane node ตัวแรก
kubeadm init \
  --control-plane-endpoint=cluster-endpoint \
  --pod-network-cidr=192.168.0.0/16 \
  --upload-certs

อธิบายจาก flag

  1. --control-plane-endpoint=cluster-endpoint — ชื่อ/IP ของ load balancer ที่วางแผนไว้ในบทที่ 3 บังคับให้ kubeadm เขียน endpoint นี้ลงใน kubeconfig ทุกไฟล์ที่ generate แทนที่จะใช้ IP ของ control-plane node ตัวแรกตรง ๆ — ถ้าไม่ใส่ flag นี้ตั้งแต่ init ครั้งแรก จะย้อนกลับมาใส่ทีหลังไม่ได้ง่าย ๆ
  2. --pod-network-cidr=192.168.0.0/16 — CIDR ที่วางแผนไว้ในบทที่ 3 ต้องตรงกับค่า default ของ Calico ที่จะลงในขั้นถัดไป ถ้าเปลี่ยน CNI ต้องเช็ค default CIDR ของ CNI นั้นให้ตรงกันด้วย
  3. --upload-certs — เข้ารหัสและอัปโหลด certificate ของ control plane ขึ้น etcd ชั่วคราว เพื่อให้ control-plane node ตัวถัด ๆ ไปที่จะ join ดึง cert ชุดเดียวกันไปใช้ได้ โดยไม่ต้อง copy cert ข้ามเครื่องด้วยมือ
  4. ท้าย output จะมีคำสั่ง kubeadm join สองแบบให้ copy ไว้ — แบบสำหรับ worker กับแบบสำหรับ control-plane node เพิ่ม เก็บไว้ใช้ในขั้นถัดไป

ตั้งค่า kubeconfig ให้ user ปกติสั่ง kubectl ได้โดยไม่ต้อง sudo

control-plane node ตัวแรก
mkdir -p $HOME/.kube
sudo cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
sudo chown $(id -u):$(id -g) $HOME/.kube/config

อธิบายจาก command

  1. /etc/kubernetes/admin.conf — kubeconfig ที่ kubeadm สร้างให้อัตโนมัติหลัง init สำเร็จ มีสิทธิ์ admin เต็ม
  2. cp -i — copy มาไว้ที่ ~/.kube/config ซึ่งเป็นตำแหน่งที่ kubectl มองหา config โดย default
  3. chown — เปลี่ยนเจ้าของไฟล์จาก root มาเป็น user ปัจจุบัน จะได้ไม่ต้อง sudo kubectl ทุกครั้ง

  2. --token — token ชั่วคราวที่ใช้ยืนยันตัวตนตอน join มีอายุ 24 ชั่วโมงโดย default
  3. --discovery-token-ca-cert-hash — hash ของ CA certificate ใช้ยืนยันว่ากำลังต่อเข้า apiserver ที่ถูกต้องจริง ๆ ป้องกัน man-in-the-middle

🌐 บทที่ 6

Service มีกี่แบบ แล้ว bare-metal จะมี LoadBalancer ได้ยังไงในเมื่อไม่มี Cloud ให้เรียก

คลัสเตอร์ HA พร้อมแล้ว แอปรันอยู่ข้างในแล้ว แต่จะให้คนนอกเข้าถึงยังไง บทนี้ไล่ตั้งแต่ Service type พื้นฐาน, CoreDNS, ไปจนถึง MetalLB ที่ทำให้ type: LoadBalancer ใช้งานได้จริงบน bare-metal

🙋
นาย ข.ผมลองสร้าง Service type LoadBalancer ดูครับ แต่ EXTERNAL-IP มันค้าง <pending> ไม่ยอมขึ้น IP สักที ทั้งที่บน EKS/GKE ผมทำแบบนี้แล้วได้ IP ทันทีเลยนะ
🧑‍🏫
นาย ก.เพราะบน EKS/GKE มี cloud-controller-manager คอยสั่งสร้าง load balancer ของ cloud นั้นให้อัตโนมัติ แต่บน bare-metal ไม่มี cloud ให้สั่ง จะค้าง pending แบบนี้ตลอดไปถ้าไม่มีอะไรมาเติมช่องว่างนั้น — นี่คืองานของ MetalLB

กล่าวสั้น ๆ คือ Service คือชื่อกลางที่ Pod เปลี่ยนไปมาได้แต่ยังเข้าถึงได้เสมอ ส่วน type: LoadBalancer บน bare-metal ต้องมีตัวช่วยอย่าง MetalLB มาทำหน้าที่แทน cloud provider ถึงจะได้ IP จริง ไม่ค้าง pending

ก่อนไปเรื่อง LoadBalancer พูดถึง CoreDNS สักนิด — CoreDNS คือ DNS server ที่รันเป็น add-on มาตรฐานของคลัสเตอร์ (เป็น Pod ที่เราเห็นตอนมันค้าง Pending รอ CNI ในบทที่แล้ว) หน้าที่ของมันคือทำให้ Pod เรียก Service ด้วยชื่อ (เช่น my-service.default.svc.cluster.local) แทนที่จะต้องจำ ClusterIP เป็นตัวเลขทุกครั้ง — ทุก Service ที่สร้างจะได้ DNS record นี้ให้อัตโนมัติ ไม่ต้องตั้งค่าอะไรเพิ่ม

(b) Dockerfile แบบ 3-stage

ทีนี้พอมี standalone output แล้ว ขั้นต่อไปคือห่อมันด้วย Dockerfile ที่แยก stage ชัดเจน — stage ติดตั้ง dependency, stage build, และ stage runner ที่เอาเฉพาะของที่จำเป็นไปวิ่งจริง จะได้ image เล็กและไม่พก build tool ติดไปโดยไม่จำเป็น

Dockerfile
ARG NODE_VERSION=20-slim
FROM node:${NODE_VERSION} AS dependencies
WORKDIR /app
COPY package.json yarn.lock* package-lock.json* pnpm-lock.yaml* .npmrc* ./
RUN --mount=type=cache,target=/root/.npm \
    --mount=type=cache,target=/usr/local/share/.cache/yarn \
    --mount=type=cache,target=/root/.local/share/pnpm/store \
  if [ -f package-lock.json ]; then \
    npm ci --no-audit --no-fund; \
  elif [ -f yarn.lock ]; then \
    corepack enable yarn && yarn install --frozen-lockfile --production=false; \
  elif [ -f pnpm-lock.yaml ]; then \
    corepack enable pnpm && pnpm install --frozen-lockfile; \
  else \
    echo "No lockfile found." && exit 1; \
  fi

FROM node:${NODE_VERSION} AS builder
WORKDIR /app
COPY --from=dependencies /app/node_modules ./node_modules
COPY . .
ENV NODE_ENV=production
RUN if [ -f package-lock.json ]; then npm run build; \
  elif [ -f yarn.lock ]; then corepack enable yarn && yarn build; \
  elif [ -f pnpm-lock.yaml ]; then corepack enable pnpm && pnpm build; \
  else echo "No lockfile found." && exit 1; fi

FROM node:${NODE_VERSION} AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=3000
ENV HOSTNAME="0.0.0.0"
COPY --from=builder --chown=node:node /app/public ./public
RUN mkdir .next && chown node:node .next
COPY --from=builder --chown=node:node /app/.next/standalone ./
COPY --from=builder --chown=node:node /app/.next/static ./.next/static
USER node
EXPOSE 3000
CMD ["node", "server.js"]

อธิบายจาก code

  1. stage dependencies — ติดตั้ง package ตาม lockfile ที่เจอ (npm/yarn/pnpm) พร้อม cache mount ของ BuildKit เพื่อให้ rebuild ครั้งถัดไปเร็วขึ้น (ต้องเปิด BuildKit ก่อนถึงจะใช้ --mount=type=cache ได้)
  2. stage builder — copy node_modules จาก stage ก่อนหน้า แล้ว copy source ทั้งหมดมา npm run build ซึ่งจะสร้าง .next/standalone และ .next/static ตามที่คุยไปใน (a)
  3. stage runner — เป็น stage สุดท้ายที่ image จริงจะถูกสร้างจาก stage นี้เท่านั้น มันไม่พก devDependency หรือ source code เต็มไปด้วย เอาเฉพาะ public/, .next/standalone, .next/static เท่านั้น
  4. COPY --chown=node:node — copy พร้อม set owner เป็น user node (ไม่ใช่ root) ตั้งแต่ตอน copy เลย จะได้ไม่ต้อง chown ทีหลังซึ่งเปลือง layer
  5. USER node — รัน process ด้วย non-root user ตาม security best practice ของ container
  6. CMD ["node", "server.js"] — รัน server.js ที่ standalone output สร้างให้ แทนที่จะเป็น next start

(c) Deployment — สั่ง Kubernetes ว่า Pod ต้องวิ่งยังไง

มี image พร้อมแล้ว ทีนี้ต้องบอก Kubernetes ว่าจะรัน image นี้กี่ตัว ใช้ resource เท่าไหร่ เช็ค health ยังไง และเอา config/secret เข้ามาทางไหน — นี่คือ field ที่ทีมส่วนใหญ่มักจะลืมใส่ตอนเริ่มต้น แล้วมาเจอปัญหาตอน production จริง

⚠️ YAML นี้เป็นของที่เราประกอบเอง

YAML ด้านล่างเป็น composite ที่บทความนี้ประกอบขึ้นเอง โดยรวม field ที่ถูก document แยกกันในหลายหน้าของ Kubernetes docs (Deployment fields, probe fields, resources fields, envFrom pattern) เข้าเป็น manifest เดียวที่ใช้งานได้จริง — ไม่ใช่การ quote จากหน้า doc เดียวแบบคำต่อคำ แนะนำให้เทียบกับ API reference ของแต่ละ field อีกทีก่อนเอาไปใช้ production จริง

deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nextjs-app
  labels:
    app: nextjs-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nextjs-app
  template:
    metadata:
      labels:
        app: nextjs-app
    spec:
      containers:
      - name: nextjs-app
        image: myregistry/nextjs-app:1.0.0
        ports:
        - containerPort: 3000
        resources:
          requests:
            cpu: "100m"
            memory: "128Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"
        readinessProbe:
          httpGet:
            path: /api/health
            port: 3000
          initialDelaySeconds: 5
          periodSeconds: 10
          timeoutSeconds: 2
          failureThreshold: 3
        livenessProbe:
          httpGet:
            path: /api/health
            port: 3000
          initialDelaySeconds: 15
          periodSeconds: 20
          timeoutSeconds: 2
          failureThreshold: 3
        envFrom:
        - configMapRef:
            name: nextjs-config
        - secretRef:
            name: nextjs-secrets

อธิบายจาก code

  1. spec.replicas: 3 — บอกให้รัน Pod พร้อมกัน 3 ตัว ป้องกันจุดเดียวล่มทั้งระบบ (single point of failure)
  2. selector.matchLabels / template.metadata.labels — ต้องตรงกันเป๊ะ Deployment ใช้ label นี้ค้นหาว่า Pod ตัวไหนเป็นของตัวเอง
  3. resources.requests — ปริมาณ CPU/memory ขั้นต่ำที่ scheduler ใช้ตัดสินใจว่าจะวาง Pod ลง node ไหน
  4. resources.limits — เพดานสูงสุดที่ container ห้ามใช้เกิน ถ้าใช้ memory เกิน limit จะโดน OOMKill
  5. readinessProbe — เช็คว่า Pod พร้อมรับ traffic หรือยัง ถ้ายังไม่ผ่าน (status ไม่อยู่ในช่วง 200-399) Service จะไม่ส่ง traffic เข้ามาที่ Pod นี้
  6. livenessProbe — เช็คว่า process ยังมีชีวิตอยู่ไหม ถ้า fail ครบ failureThreshold ครั้ง Kubernetes จะ restart container ให้อัตโนมัติ
  7. initialDelaySeconds — เวลารอก่อนเริ่มเช็ค probe ครั้งแรก กันไม่ให้ app ที่ยัง boot ไม่เสร็จโดนตัดสินว่า fail ตั้งแต่ยังไม่ทันเริ่ม
  8. envFrom — ดึง key ทั้งหมดจาก ConfigMap และ Secret มาเป็น environment variable ให้ container โดยอัตโนมัติ (รายละเอียดในหัวข้อ (e))

(d) Service + Ingress — เปิดทางให้ traffic เข้าถึง Pod

Pod แต่ละตัวมี IP ที่เปลี่ยนไปเรื่อยๆ ตาม restart ดังนั้นเราต้องมี Service เป็นจุดคงที่ให้ traffic เข้ามาหา แล้วมี Ingress เป็นประตูหน้าบ้านที่ map domain/path เข้า Service อีกที

service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nextjs-app
spec:
  selector:
    app: nextjs-app
  ports:
  - port: 80
    targetPort: 3000
ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: minimal-ingress
spec:
  ingressClassName: nginx-example
  rules:
  - http:
      paths:
      - path: /testpath
        pathType: Prefix
        backend:
          service:
            name: test
            port:
              number: 80

อธิบายจาก code

  1. Service selector.app: nextjs-app — ต้องตรงกับ label ของ Pod ใน Deployment ไม่งั้น Service จะหา Pod ปลายทางไม่เจอเลย (endpoint ว่างเปล่า)
  2. port vs targetPort — port: 80 คือ port ที่ Service เปิดให้เรียกจากภายใน cluster ส่วน targetPort: 3000 คือ port จริงที่ container ฟังอยู่ (ตรงกับ containerPort ใน Deployment)
  3. Ingress ingressClassName: nginx-example — บอกว่าจะให้ Ingress controller ตัวไหนเป็นคนดูแล rule นี้ ถ้าไม่ใส่ต้องมี default IngressClass ตั้งไว้ในคลัสเตอร์อยู่แล้ว
  4. pathType: Prefix — จับคู่ path แบบ prefix เช่น /testpath จะครอบคลุม /testpath/anything ด้วย
  5. backend.service.name / port.number — ชี้ไปที่ Service ที่สร้างไว้ด้านบน ต้องเปลี่ยนชื่อให้ตรงกับ Service จริง (ในตัวอย่าง official เป็นชื่อ test ต้องแก้เป็น nextjs-app เวลาใช้จริง)

💡 Ingress vs Gateway API

Kubernetes docs เวอร์ชันปัจจุบันแนะนำให้โปรเจกต์ใหม่พิจารณาใช้ Gateway API แทน Ingress เพราะยืดหยุ่นกว่าและเป็นทิศทางที่ ecosystem กำลังไป แต่บทความนี้ยังใช้ Ingress เป็นหลักเพราะ production setup ส่วนใหญ่ในปัจจุบันยังวิ่งบน Ingress อยู่ และ ecosystem ของ controller (nginx, traefik ฯลฯ) ยังโตกว่ามาก — ถ้าเริ่มโปรเจกต์ใหม่จริงๆ แนะนำให้อ่านเรื่อง Gateway API เพิ่มเติมประกอบการตัดสินใจ

(e) ConfigMap และ Secret — แยก config ออกจาก image

อย่า hardcode ค่าอย่าง API URL หรือ database connection string ลงใน image เด็ดขาด เพราะทุกครั้งที่ค่าเปลี่ยนจะต้อง build image ใหม่ ConfigMap กับ Secret คือ object ของ Kubernetes ที่ให้เราแยกค่าพวกนี้ออกมา แล้วฉีดเข้า Pod ตอน runtime แทน

configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nextjs-config
data:
  NEXT_PUBLIC_API_URL: "https://api.example.com"
  NODE_ENV: "production"

  name: nextjs-secrets
type: Opaque

      name: nextjs-config

อธิบายจาก code

  1. ConfigMap data — เก็บค่าที่ไม่ sensitive เช่น URL หรือ flag ต่างๆ เป็น plain text ธรรมดา
  2. Secret type: Opaque — ค่า default ของ Secret ทั่วไปที่ไม่มีรูปแบบเฉพาะ (ต่างจาก type อื่นเช่น kubernetes.io/tls)
  3. Secret data.DATABASE_URL — ค่าต้อง encode เป็น base64 เสมอ (ในตัวอย่างคือ postgres://user:pass@db/name ที่ผ่าน base64 มาแล้ว) — base64 ไม่ใช่การเข้ารหัส เป็นแค่ encoding ใครมี access อ่าน Secret ก็ decode กลับเป็น plain text ได้ทันที ต้องพึ่ง RBAC หรือเครื่องมือเช่น sealed-secrets/external-secrets ช่วยเพิ่มชั้นความปลอดภัย
  4. envFrom.configMapRef.name / secretRef.name — สั่งให้ทุก key ใน ConfigMap และ Secret ถูกแปลงเป็น environment variable ให้ container โดยอัตโนมัติ ไม่ต้อง list ทีละตัวด้วย env

(f) HorizontalPodAutoscaler — scale ตาม load อัตโนมัติ

ตั้ง replicas: 3 แบบตายตัวไปตลอดก็ใช้ได้ในวันธรรมดา แต่พอ traffic พุ่งตอน campaign หรือช่วง peak จะไม่พอ HorizontalPodAutoscaler (HPA) คือ object ที่คอย monitor resource usage แล้วเพิ่ม/ลดจำนวน replicas ให้เองอัตโนมัติ

terminal
# สร้าง HPA แบบ imperative command เดียวจบ
kubectl autoscale deployment nextjs-app --cpu=50% --min=2 --max=10

💡 YAML ด้านล่างมาจาก generation ไม่ใช่ verbatim quote 100%

รูปทรงของ metrics: block ด้านล่างตรงกับ public API shape ของ autoscaling/v2 ที่รู้จักกันทั่วไป แต่ในการค้นคว้าครั้งนี้ไม่ได้ capture มาแบบคำต่อคำจากหน้า doc เดียว วิธีที่แนะนำจริงๆ คือรัน kubectl autoscale ตามคำสั่งด้านบนก่อน แล้ว export YAML จริงออกมาด้วย kubectl get hpa nextjs-app -o yaml เพื่อความชัวร์ 100% ว่า field ตรงกับเวอร์ชันคลัสเตอร์ที่ใช้อยู่

hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nextjs-app
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nextjs-app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50

อธิบายจาก code

  1. kubectl autoscale deployment nextjs-app --cpu=50% --min=2 --max=10 — วิธี imperative สร้าง HPA ให้เล็งไปที่ Deployment ชื่อ nextjs-app โดยตรง สั้นและเร็วเหมาะกับตอน demo หรือ debug
  2. scaleTargetRef — ชี้ว่า HPA นี้จะควบคุม replicas ของ workload ตัวไหน (ในที่นี้คือ Deployment ชื่อ nextjs-app ตัวเดียวกับที่เขียนไว้ในหัวข้อ (c))
  3. minReplicas / maxReplicas — กำหนดขอบเขตล่าง-บนของจำนวน Pod ห้ามลดต่ำกว่า 2 และห้ามเกิน 10 ไม่ว่า load จะขึ้นแรงแค่ไหน
  4. metrics[0].resource.name: cpu — ใช้ CPU utilization เป็นตัวชี้วัดหลักในการตัดสินใจ scale (ยังผูก metric อื่นเพิ่มได้ เช่น memory หรือ custom metric)
  5. target.averageUtilization: 50 — ถ้า CPU เฉลี่ยของ Pod ทั้งหมดเกิน 50% ของ resources.requests.cpu ที่ตั้งไว้ใน Deployment, HPA จะเพิ่ม replicas ให้เอง

💡 HPA ต้องพึ่ง resources.requests

HPA คำนวณ % utilization เทียบกับค่า resources.requests.cpu ที่ตั้งไว้ในหัวข้อ (c) ถ้าลืมตั้ง requests ไว้ HPA จะไม่มีฐานให้คำนวณ แล้วจะ scale ไม่ทำงานเลย นี่คือเหตุผลที่ resources กับ HPA ต้องมองคู่กันเสมอ

Helm — package manager ของ Kubernetes

Helm มองทุกชุด manifest เป็น "chart" หนึ่งชุด แล้วให้ values file เป็นตัวกำหนดค่าที่ต่างกันในแต่ละ environment โดยไม่ต้องแก้ template

helm create nextjs-app

อธิบายจาก code

  1. Chart.yaml — metadata ของ chart field ที่ต้องมีคือ apiVersion, name, version ส่วน appVersion, description, dependencies, maintainers เป็น optional
  2. values.yaml — ค่า default ทั้งหมดของ chart เช่น จำนวน replicas, image tag, resources — เวลา deploy environment ไหนก็ override ค่าพวกนี้ผ่าน -f หรือ --set
  3. charts/ — เก็บ sub-chart (dependency chart อื่นที่ chart นี้พึ่งพา)
  4. templates/ — เก็บไฟล์ YAML แบบ template (Deployment, Service, Ingress ฯลฯ จากบทที่แล้ว) ที่แทรก {{ .Values.xxx }} เพื่อดึงค่าจาก values.yaml
  5. โครงสร้างแบบเต็มของ chart จริงยังมี LICENSE, README.md, values.schema.json (validate ค่า values), crds/ และ templates/NOTES.txt (ข้อความแสดงหลัง install สำเร็จ) เพิ่มได้ตามต้องการ
terminal
# pattern มาตรฐานสำหรับ deploy ผ่าน CI/CD — idempotent
# ถ้ายังไม่เคย install จะ install ให้ ถ้ามีอยู่แล้วจะ upgrade ให้
helm upgrade nextjs-app ./nextjs-app --install

# ตัวอย่างอื่นจาก doc — ระบุ values file หรือ --set ค่าเดี่ยวๆ
helm upgrade -f myvalues.yaml redis ./redis
helm upgrade --set foo=bar redis ./redis

อธิบายจาก code

  1. helm upgrade [RELEASE] [CHART] --install — flag --install (หรือ -i) ทำให้คำสั่งนี้ idempotent: "ถ้ายังไม่มี release ชื่อนี้อยู่ ให้ install ใหม่แทน" นี่คือเหตุผลที่ pipeline CI/CD แทบทุกที่ใช้คำสั่งเดียวนี้ทั้งตอน deploy ครั้งแรกและ deploy ซ้ำ ไม่ต้องเขียน logic แยก install/upgrade เอง
  2. -f myvalues.yaml — override ค่าจาก values.yaml เริ่มต้นด้วยไฟล์ values แยกต่างหาก เหมาะกับ environment ที่มีค่าต่างกันเยอะ
  3. --set foo=bar — override ค่าเดี่ยวๆ ผ่าน command line เหมาะกับตอน debug หรือค่าที่เปลี่ยนบ่อยเช่น image tag

💡 Helm 4 / OCI vs classic repo add

เอกสารของ Helm ปัจจุบันปรับ quickstart หลักไปใช้ Helm 4 กับ OCI registry เป็นทางหลักแล้ว (เช่น helm install my-podinfo oci://ghcr.io/...) แต่ flow แบบคลาสสิกที่ใช้ helm repo add ตามด้วย helm install จาก repo index ก็ยังใช้งานได้และยัง document อยู่เต็มรูปแบบ บทความนี้เลือกอธิบายด้วย flow แบบ repo-based คลาสสิกเพราะยังเป็นวิธีที่ production ทีมส่วนใหญ่ใช้กันอยู่จริงในปัจจุบัน ถ้าทีมไหนพร้อมย้ายไป OCI registry ก็ทำได้เลย concept เรื่อง chart/values/release เหมือนกันทั้งหมด

    includeSelectors: true
commonAnnotations:
  oncallPager: "800-555-1212"
resources:
- ../../base

อธิบายจาก code

  1. base/kustomization.yaml → resources — list ไฟล์ YAML ดิบๆ ที่เป็นต้นแบบร่วมของทุก environment (deployment.yaml, service.yaml จากบทที่แล้ว)
  2. configMapGenerator — สร้าง ConfigMap ให้อัตโนมัติจากไฟล์ (application.properties) แทนที่จะเขียน ConfigMap YAML เองแบบบทที่แล้ว
  3. overlay resources: ../../base — overlay ไม่ copy YAML ของ base มาซ้ำ แค่ "อ้างอิง" กลับไปที่ base directory แล้ว patch ทับเฉพาะส่วนที่ต่าง
  4. namespace: production — inject namespace ให้ทุก resource ใน base โดยไม่ต้องแก้ไฟล์ต้นฉบับ
  5. namePrefix: prod- — เติม prefix หน้าชื่อทุก resource อัตโนมัติ เช่น Deployment ชื่อ nextjs-app จะกลายเป็น prod-nextjs-app
  6. labels + includeSelectors: true — เติม label env: production ให้ทุก resource และให้ label นี้มีผลกับ selector ด้วย (ไม่ใช่แค่ metadata เฉยๆ)
  7. commonAnnotations — เติม annotation เดียวกันให้ทุก resource เช่น เบอร์ oncall ของทีมที่ดูแล environment นี้
terminal
# ดูผลลัพธ์ YAML ที่ประกอบเสร็จแล้วก่อน apply จริง
kubectl kustomize overlays/production

# apply overlay เข้าคลัสเตอร์ตรงๆ (มี kustomize built-in ใน kubectl ตั้งแต่ v1.14)
kubectl apply -k overlays/production

อธิบายจาก code

  1. kubectl kustomize <dir> — render YAML สุดท้ายออกมาดูเฉยๆ ไม่ apply ใช้ตอน review หรือ debug ว่า base + overlay รวมกันแล้วหน้าตาเป็นยังไง
  2. kubectl apply -k <dir> — flag -k (หรือ --kustomize) สั่งให้ kubectl render kustomization ในไดเรกทอรีนั้นแล้ว apply เข้าคลัสเตอร์ทันที ไม่ต้องติดตั้ง binary kustomize แยกเลยเพราะฝังอยู่ใน kubectl ตั้งแต่ v1.14 แล้ว (แม้ standalone CLI kustomize build ก็ยังนิยมใช้กันเหมือนกัน โดยเฉพาะตอนต้องการ feature เวอร์ชันใหม่กว่าที่ฝังใน kubectl)

💡 เลือกยังไงดี

ถ้าทีมต้อง publish chart ให้คนอื่นใช้ต่อ หรือต้องการ release history/rollback ในตัว — Helm ตอบโจทย์กว่า ถ้าทีมอยากให้ YAML อ่านง่าย ไม่มี syntax แปลกปลอม และแค่ต้องการ manage ความต่างระหว่าง dev/staging/production — Kustomize เรียบง่ายกว่าเยอะ หลายทีมจริงๆ ใช้ทั้งคู่ผสมกัน เช่น Helm สำหรับ chart จาก third-party (Redis, Postgres) และ Kustomize สำหรับ manifest ของ app ตัวเองที่เขียนเอง

Backup กับ Disaster Recovery: ทำไม Backup ที่ไม่เคย Restore ถึงยังไม่ใช่ Backup จริง

บทนี้พาไปดูการทำ backup สองชั้นที่ production cluster ทุกตัวต้องมี — ชั้น workload ด้วย Velero และชั้น control plane ด้วย etcd snapshot — พร้อมพาไปซ้อม restore จริงๆ เพราะไฟล์ backup ที่ไม่เคยถูกเอามากู้คืนสักครั้ง คือความเสี่ยงที่ซ่อนอยู่โดยไม่มีใครรู้

🧑‍💻
สมชายนาย ก. ครับ... ผมเผลอรัน kubectl delete namespace production ผิด environment ตอนนี้ Deployment, ConfigMap, Secret หายหมดเลยครับ แล้วเราไม่มี backup เก็บไว้เลย
🧑‍🏫
นาย ก....เห็นภาพเลยใช่ไหมครับ ว่าทำไมทุก production cluster ต้องมี Velero ติดตั้งไว้ตั้งแต่วันแรก ไม่ใช่ค่อยมาติดตั้งตอนพังแล้ว วันนี้เรามานั่งวางระบบ backup ที่กู้คืนได้จริงกัน

กล่าวสั้นๆ คือ Velero ดูแล backup/restore ระดับ Kubernetes object และ Persistent Volume ส่วน etcd snapshot ดูแล backup ระดับ state ทั้งคลัสเตอร์ที่ control plane เก็บไว้ — เป็นคนละชั้นกัน และ production cluster ต้องมีทั้งคู่ ขาดตัวใดตัวหนึ่งไม่ได้

หัวข้อVeleroetcd snapshot
ระดับที่ backupNamespace / object level (Deployment, Service, PVC, ...)Whole cluster state ที่เก็บใน etcd ทั้งหมด
กู้คืนอะไรได้แอปหรือ namespace ที่ถูกลบ/พังเฉพาะจุดทั้งคลัสเตอร์ตอน control plane พังหรือ etcd corrupt
เหมาะกับสถานการณ์ลบ namespace ผิด, ต้อง migrate namespace ไป cluster ใหม่etcd disk พัง, upgrade แล้วพัง, ต้อง rebuild control plane
เก็บไว้ที่ไหนObject storage ที่ compatible กับ S3 (MinIO, Ceph RGW, Wasabi ฯลฯ)ไฟล์ .db บน disk แยกต่างหาก (ควรก็อปออกไปเก็บนอกเครื่อง)

ติดตั้ง Velero เข้ากับ S3-compatible storage

ทีนี้มาเริ่มจากชั้นแรกกันก่อน — Velero ใช้ object storage เป็นที่เก็บ backup ซึ่งไม่จำเป็นต้องเป็น AWS S3 จริงก็ได้ ใช้ MinIO, Ceph RGW หรือ Wasabi แทนก็ได้ทั้งนั้น เพราะ Velero ไม่มี provider แยกสำหรับ "generic S3" — ทุกตัวที่พูด S3 API ได้ ให้ใช้ --provider aws ร่วมกับ velero-plugin-for-aws แล้วชี้ endpoint เอาเองผ่าน s3Url กับ s3ForcePathStyle

credentials-velero
[default]
aws_access_key_id = minio
aws_secret_access_key = minio123

อธิบายจาก code

  1. รูปแบบไฟล์ — เป็น INI format แบบเดียวกับ AWS credentials file เพราะ Velero ใช้ velero-plugin-for-aws คุยกับ storage ไม่ว่า backend จริงจะเป็น MinIO หรืออะไรก็ตาม
  2. aws_access_key_id / aws_secret_access_key — ในตัวอย่างนี้ใช้ค่า default ของ MinIO (minio / minio123) ใน production จริงต้องเปลี่ยนเป็น credential ของ storage ที่ใช้งานจริง
terminal
velero install \
    --provider aws \
    --plugins velero/velero-plugin-for-aws:v1.2.1 \
    --bucket velero \
    --secret-file ./credentials-velero \
    --use-volume-snapshots=false \
    --backup-location-config region=minio,s3ForcePathStyle="true",s3Url=http://minio.velero.svc:9000

อธิบายจาก code

  1. --provider aws — ใช้ provider นี้เสมอสำหรับ storage backend ที่พูด S3 API แม้จะไม่ใช่ AWS จริงก็ตาม
  2. --plugins velero/velero-plugin-for-aws:v1.2.1 — เวอร์ชันของ plugin ต้องเข้าคู่กับเวอร์ชัน Velero ที่ติดตั้ง ถ้า mismatch กันบ่อยครั้งจะ backup ไม่ผ่านแบบเงียบๆ
  3. --bucket velero — ชื่อ bucket ปลายทางที่สร้างไว้ล่วงหน้าใน storage
  4. --secret-file ./credentials-velero — ไฟล์ credential ที่เพิ่งสร้างด้านบน
  5. --use-volume-snapshots=false — ปิด native volume snapshot เพราะ MinIO ไม่รองรับ CSI snapshot ในตัวอย่างนี้ (ถ้า storage backend รองรับจริงค่อยเปิด)
  6. --backup-location-config — ระบุ s3ForcePathStyle="true" และ s3Url เพื่อบอก plugin ว่านี่คือ S3-compatible endpoint ไม่ใช่ AWS S3 จริง

💡 เวอร์ชัน plugin ต้องคู่กับเวอร์ชัน Velero

ก่อนติดตั้งทุกครั้ง เช็ก compatibility matrix ของ velero-plugin-for-aws กับ Velero เวอร์ชันที่ใช้เสมอ ใส่เวอร์ชันผิดคู่กัน backup อาจสร้างไม่สำเร็จโดยไม่มี error ที่ชัดเจน

สร้าง Backup และตั้ง Schedule

ติดตั้งเสร็จแล้ว ทีนี้มาสร้าง backup แบบ manual ครั้งแรกก่อน แล้วค่อยตั้งให้รันอัตโนมัติทุกวัน

terminal
# สร้าง backup แบบ manual ครั้งเดียว แล้วรอจนเสร็จ
velero backup create my-backup --include-namespaces production --wait

# ตั้ง schedule ให้รันทุกวันตอนตี 3
velero schedule create daily-backup --schedule="0 3 * * *"

อธิบายจาก code

  1. velero backup create my-backup — สร้าง backup ชื่อ my-backup
  2. --include-namespaces production — จำกัดขอบเขต backup ให้เอาเฉพาะ namespace production ไม่ backup ทั้งคลัสเตอร์ทุกครั้งโดยไม่จำเป็น
  3. --wait — ให้ terminal ค้างรอจน backup เสร็จจริง แทนที่จะ return ทันทีแบบ async
  4. velero schedule create daily-backup — สร้าง schedule ชื่อ daily-backup ที่จะสร้าง backup ใหม่ตาม cron expression ให้อัตโนมัติ ไม่ต้องมานั่งรันมือทุกวัน
  5. --schedule="0 3 * * *" — cron expression มาตรฐาน หมายถึงรันทุกวันตอนตี 3
🙋‍♂️
นาย ข.งั้นตอนนี้เรามี backup รันทุกวันแล้ว จบเลยใช่ไหมครับ?
🧑‍🏫
นาย ก.ยังไม่จบครับ นี่แหละจุดที่คนพลาดกันบ่อยที่สุด — backup ที่ไม่เคยถูกเอามาลอง restore สักครั้ง ก็เหมือนไม่มี backup เพราะเราไม่รู้เลยว่าไฟล์ backup มันใช้กู้คืนได้จริงหรือเปล่า จนกว่าจะถึงวันที่ต้องใช้จริง ซึ่งอาจสายเกินไป
🧑‍💻
สมชายอ๋อ... งั้นเรามาลองซ้อม restore กันเลยดีกว่าครับ จะได้รู้ว่าถ้าวันจริงมาถึง เรากู้ทันไหม
terminal
# สร้าง restore จาก backup ที่มีอยู่ แล้วรอจนเสร็จ
velero restore create my-restore --from-backup my-backup --wait

# ดูรายละเอียดว่า restore สำเร็จไหม มี object ไหนติดปัญหาบ้าง
velero restore describe my-restore

# ดู log แบบละเอียด ถ้าต้องไล่ debug
velero restore logs my-restore

อธิบายจาก code

  1. velero restore create my-restore --from-backup my-backup — สั่งกู้คืนจาก backup ชื่อ my-backup เข้าเป็น restore job ชื่อ my-restore
  2. --wait — เหมือนเดิม รอจนกู้เสร็จจริงก่อนคืน terminal
  3. velero restore describe my-restore — เช็กสถานะว่า Completed หรือ PartiallyFailed แล้วดูว่า warning/error เกิดกับ resource ไหนบ้าง — ขั้นตอนนี้ห้ามข้าม เพราะ restore อาจ "เสร็จ" แต่ไม่ได้แปลว่า "ครบ"
  4. velero restore logs my-restore — ใช้เวลาต้องไล่ debug ว่าทำไม object บางตัวถึง restore ไม่ผ่าน
checklist ซ้อม restore ที่ทีมควรทำอย่างน้อยไตรมาสละครั้ง

restore ใส่ namespace ทดสอบแยกต่างหาก (ไม่ใช่ทับของจริง) → เช็กว่า Deployment/Service/ConfigMap/Secret มาครบ → เช็กว่าแอปรันขึ้นจริง ไม่ใช่แค่ resource ถูกสร้าง → จับเวลาทั้งกระบวนการไว้เป็น RTO จริงที่วัดได้ ไม่ใช่ตัวเลขที่เดาไว้ในเอกสาร

แล้วถ้า control plane ทั้งคลัสเตอร์พังล่ะ Velero ช่วยได้ไหม

คำตอบคือไม่ครับ — Velero backup ที่ระดับ Kubernetes object ผ่าน API server เท่านั้น ถ้า etcd ซึ่งเป็นที่เก็บ state ทั้งหมดของคลัสเตอร์พังหรือ corrupt ไปเลย ต้องมี etcd snapshot แยกต่างหากถึงจะกู้คลัสเตอร์กลับมาได้ ในคลัสเตอร์ที่ตั้งด้วย kubeadm การ snapshot ต้องยิงตรงผ่าน etcdctl พร้อมชี้ TLS certificate ของ etcd เอง

terminal
ETCDCTL_API=3 etcdctl --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  snapshot save /backup/etcd-snapshot.db

อธิบายจาก code

  1. ETCDCTL_API=3 — บังคับใช้ etcdctl API v3 ซึ่งเป็น API ปัจจุบันที่ etcd ใช้จริง
  2. --endpoints=https://127.0.0.1:2379 — พอร์ตมาตรฐานของ etcd บน control plane node ที่รันคำสั่งนี้
  3. --cacert / --cert / --key — path มาตรฐานของ certificate ที่ kubeadm สร้างไว้ให้ตอน bootstrap คลัสเตอร์ ต้องชี้ทั้งสามตัวนี้ให้ครบ ไม่งั้น etcdctl จะต่อ TLS ไม่ผ่าน
  4. snapshot save /backup/etcd-snapshot.db — คำสั่งจริงที่สร้างไฟล์ snapshot ควรก็อปไฟล์นี้ออกไปเก็บนอกเครื่อง control plane ทันที เพราะถ้า disk เครื่องนี้พังพร้อมกับ etcd ไฟล์ backup ก็หายไปด้วย
terminal
etcdutl --data-dir /var/lib/etcd-restore snapshot restore /backup/etcd-snapshot.db

อธิบายจาก code

  1. etcdutl — เครื่องมือที่แนะนำให้ใช้ในปัจจุบันสำหรับ restore snapshot (ดู callout ด้านล่างว่าทำไมไม่ใช้ etcdctl)
  2. --data-dir /var/lib/etcd-restore — restore ไปยัง data directory ใหม่ ไม่ทับของเดิมตรงๆ เพื่อกันพลาด แล้วค่อยสลับ path ทีหลัง
  3. snapshot restore /backup/etcd-snapshot.db — ไฟล์ snapshot ที่ save ไว้จากขั้นตอนก่อนหน้า
  4. ก่อนรันจริง — ต้อง stop kube-apiserver ทุกตัวก่อน และต้อง restore state ให้ครบทุก etcd instance ในคลัสเตอร์ (ถ้าเป็น HA control plane หลายโหนด) ก่อนจะ restart อะไรทั้งนั้น ไม่งั้น state จะไม่ตรงกันระหว่างโหนด

⚠️ etcdctl snapshot restore เลิกใช้แล้ว

ตั้งแต่ etcd v3.5.x เป็นต้นมา คำสั่ง etcdctl snapshot restore ถูก deprecate แล้ว เครื่องมือที่แนะนำให้ใช้แทนคือ etcdutl snapshot restore ตามตัวอย่างด้านบน ถ้าเอกสารเก่าที่เจอยังสอน etcdctl snapshot restore อยู่ ให้เช็กเวอร์ชัน etcd ของคลัสเตอร์ก่อนว่าคำสั่งนั้นยังใช้ได้จริงไหม

ทีนี้เราวางระบบ backup ครบสองชั้นแล้ว — Velero คอยดูแลระดับ workload พร้อม schedule อัตโนมัติและซ้อม restore แล้วจริง ส่วน etcd snapshot คอยดูแลระดับ control plane ทั้งคลัสเตอร์ แต่มี backup ดีแค่ไหนก็ช่วยไม่ได้ ถ้าไม่มีใครรู้ตัวว่าระบบกำลังพังอยู่ตอนนี้เลย — บทต่อไปเรามาทำให้คลัสเตอร์ "พูดได้" ด้วย monitoring stack กัน

ให้ Prometheus scrape /metrics ของแอป Next.js ผ่าน ServiceMonitor

ติดตั้ง stack เสร็จแล้ว Prometheus จะเก็บ metrics ของคลัสเตอร์ให้อัตโนมัติ แต่ยังไม่รู้จักแอปของเราเอง ต้องประกาศผ่าน ServiceMonitor ซึ่งเป็น CRD ที่ Prometheus Operator คอยฟังอยู่

servicemonitor-nextjs.yaml
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: nextjs-app
  labels:
    release: monitoring
spec:
  selector:
    matchLabels:
      app: nextjs-app
  endpoints:
  - port: web
    path: /metrics

อธิบายจาก code

  1. labels.release: monitoring — kube-prometheus-stack ตั้ง Prometheus ให้เลือกเฉพาะ ServiceMonitor ที่มี label ตรงกับชื่อ Helm release เท่านั้น ถ้าใส่ label นี้ผิดหรือลืมใส่ Prometheus จะมองไม่เห็น ServiceMonitor นี้เลย
  2. spec.selector.matchLabels.app: nextjs-app — เลือก Service ที่มี label app: nextjs-app เป็นเป้าหมายให้ scrape ต้องตรงกับ label บน Service ของแอปจริง
  3. endpoints.port: web — ชื่อ port (ไม่ใช่เลขพอร์ต) ที่ประกาศไว้ใน spec ของ Service ต้องตั้งชื่อ port ตรงกันทั้งสองฝั่ง
  4. path: /metrics — endpoint ที่แอป Next.js เปิดไว้ให้ Prometheus ดึงค่า metrics

หมายเหตุ: การใช้ path /metrics เป็น convention ที่ทั้งวงการใช้กันเป็นมาตรฐานทั่วไป ไม่ใช่ค่าที่ Prometheus บังคับตายตัว ถ้าแอปเปิด metrics endpoint ไว้ path อื่น ก็แค่เปลี่ยนค่านี้ให้ตรงกับที่แอปใช้จริง

ตั้ง Alert Rule ให้แจ้งเตือนอัตโนมัติ

มี metrics แล้ว ทีนี้มาตั้งกฎว่าเมื่อไหร่ควรแจ้งเตือน ตัวอย่างด้านล่างคือ rule แบบ native ของ Prometheus เอง — syntax แบบนี้ใช้ตรงกับไฟล์ rule ที่ Prometheus โหลดเข้าไปโดยตรง

nextjs-app-alerts.rules.yaml
groups:
- name: nextjs-app-alerts
  rules:
  - alert: InstanceDown
    expr: up == 0
    for: 5m
    labels:
      severity: page
    annotations:
      summary: "Instance {{ $labels.instance }} down"
      description: "{{ $labels.instance }} of job {{ $labels.job }} has been down for more than 5 minutes."

อธิบายจาก code

  1. groups / name: nextjs-app-alerts — จัดกลุ่ม rule ไว้ด้วยกัน ตั้งชื่อกลุ่มให้สื่อความหมายว่าเป็น alert ของแอปตัวไหน
  2. alert: InstanceDown — ชื่อ alert ที่จะไปโผล่ใน Alertmanager และ Grafana
  3. expr: up == 0 — PromQL expression ที่ตรวจว่า target ที่ Prometheus scrape อยู่หายไปหรือเปล่า (metric up เป็น 0 แปลว่า scrape ไม่ติด)
  4. for: 5m — เงื่อนไขต้องเป็นจริงต่อเนื่อง 5 นาทีก่อนถึงจะยิง alert จริง กันไม่ให้ alert ปลอมจาก network กระตุกแป๊บเดียว
  5. labels.severity: page — ใช้ routing ใน Alertmanager ว่า alert ระดับนี้ต้องปลุกคนจริงๆ ไม่ใช่แค่ log ไว้เฉยๆ
  6. annotations.summary / description — ใช้ template variable อย่าง {{ $labels.instance }} และ {{ $labels.job }} ดึงค่าจาก label จริงมาใส่ในข้อความแจ้งเตือน

💡 rule ไฟล์ตรง vs PrometheusRule CRD ไม่เหมือนกันเป๊ะ

โค้ดด้านบนคือ syntax rule-file แบบ native ของ Prometheus เอง ถ้าจะใช้กับ kube-prometheus-stack ต้องห่อ groups:/rules: ชุดเดียวกันนี้ไว้ใต้ spec: ของ CRD PrometheusRule (apiVersion: monitoring.coreos.com/v1, kind: PrometheusRule) ตามธรรมเนียมที่ใช้กันทั่วไปในระบบนิเวศ Prometheus Operator — แต่รูปแบบ YAML ที่ห่อ CRD แบบเป๊ะๆ นี้ ทีมยังไม่ได้ตรวจสอบคำต่อคำกับเอกสารต้นทางแยกต่างหาก ก่อนใช้จริงแนะนำให้ตรวจ field ให้ตรงกับเวอร์ชัน Prometheus Operator ที่ติดตั้งอยู่อีกที

Grafana Dashboard โหลดอัตโนมัติผ่าน ConfigMap Sidecar

Load Testing กับ Conformance Testing: มั่นใจได้ยังไงว่าคลัสเตอร์พร้อม Production จริง

บทนี้ว่าด้วยการยิง load test ด้วย k6 เข้า Ingress ของแอป Next.js จริง การตรวจ readiness/liveness probe เป็นส่วนหนึ่งของ deploy validation และแนะนำให้รู้จัก sonobuoy สำหรับตรวจว่าคลัสเตอร์เป็นไปตามมาตรฐาน Kubernetes conformance

🙋‍♂️
นาย ข.ผม deploy เสร็จแล้ว ผ่าน CI ทุก stage เขียว หมดแล้วครับ ขึ้น production ได้เลยใช่ไหม?
🧑‍🏫
นาย ก.CI เขียวแปลว่า build ผ่านกับ test unit ผ่านครับ แต่ยังไม่รู้เลยว่าถ้ามีคนเข้าเว็บพร้อมกัน 500 คน ระบบจะรับไหวไหม นี่แหละหน้าที่ของ load test

กล่าวสั้นๆ คือ load testing ตอบคำถามว่า "ระบบของเรารับ traffic จริงไหวไหม" ส่วน conformance testing ตอบคำถามว่า "คลัสเตอร์นี้เป็นไปตามมาตรฐาน Kubernetes จริงไหม" เป็นคนละคำถามกัน และทั้งคู่ต่างจาก readiness/liveness probe ที่ตอบว่า "pod นี้พร้อมรับ traffic ตอนนี้ไหม" — ต้องเช็กทั้งสามระดับก่อนมั่นใจว่าคลัสเตอร์พร้อม production จริง

ยิง Load Test เข้า Ingress ด้วย k6

เริ่มจากสคริปต์ k6 ง่ายๆ ก่อน — จำลอง user 10 คนยิง request เข้า endpoint จริงของแอปพร้อมกันต่อเนื่อง 30 วินาที

script.js
import http from 'k6/http';
import { sleep } from 'k6';

export const options = {
  vus: 10,
  duration: '30s',
};

export default function () {
  http.get('https://nextjs-app.example.com/');
  sleep(1);
}

อธิบายจาก code

  1. import http from 'k6/http' — module สำหรับยิง HTTP request ของ k6
  2. options.vus: 10 — จำนวน virtual user ที่จำลองพร้อมกัน (VU = virtual user)
  3. options.duration: '30s' — ระยะเวลาที่รัน test ต่อเนื่อง
  4. export default function () — โค้ดในฟังก์ชันนี้คือสิ่งที่ virtual user แต่ละตัววนรันซ้ำตลอด duration
  5. http.get('https://nextjs-app.example.com/') — ยิง request จริงเข้า domain ที่ผูกกับ Ingress ของแอป Next.js ไม่ใช่ยิงเข้า Service ภายในคลัสเตอร์ตรงๆ เพื่อจำลอง path เดียวกับที่ user จริงเจอ (ผ่าน Ingress controller, TLS, ฯลฯ)
  6. sleep(1) — หน่วง 1 วินาทีต่อรอบ จำลอง think-time ของ user จริง ไม่ใช่ยิงรัวๆ ไม่มีหยุด
terminal
k6 run script.js

อธิบายจาก code

  1. k6 run script.js — รันสคริปต์ตามค่า options ที่ประกาศไว้ในไฟล์ จบแล้ว k6 จะสรุปผลให้ทั้ง latency (p95, p99), request/sec, error rate ทันทีที่ terminal

ทีนี้ถ้าแอปมี flow ที่สำคัญกว่าแค่หน้าแรก เช่นหน้า login ก็ทำ script แยกยิงตรง endpoint นั้นได้เลย

login-test.js
import http from 'k6/http';

export default function () {
  const url = 'https://nextjs-app.example.com/api/login';
  const payload = JSON.stringify({ email: 'test@example.com', password: 'test123' });
  const params = { headers: { 'Content-Type': 'application/json' } };
  http.post(url, payload, params);
}

อธิบายจาก code

  1. const url — ยิงตรงเข้า API route /api/login ของแอป Next.js ผ่าน Ingress endpoint เดียวกับที่ user จริงใช้
  2. JSON.stringify({...}) — สร้าง payload แบบ JSON เหมือน request จริงที่ frontend ส่งตอน login
  3. params.headers — ตั้ง Content-Type: application/json ให้ตรงกับที่ API endpoint คาดหวัง ถ้าลืมใส่ header นี้ backend อาจ parse body ไม่ออกแล้ว test fail ผิดจุด
  4. http.post(url, payload, params) — ยิง POST request จำลอง flow login จริง เหมาะกับการทดสอบ endpoint ที่มี logic หนักกว่าการ GET หน้าแรกเฉยๆ เช่นมีการ query database หรือเช็ก credential

Readiness/Liveness Probe เป็นส่วนหนึ่งของ Deploy Validation ด้วย

Load test ตอบคำถามเรื่อง throughput แต่ก่อนจะถึงจุดนั้น ต้องมั่นใจก่อนว่าแต่ละ pod เข้าสู่สถานะ ready จริงหลัง deploy ทุกครั้ง — readiness probe กันไม่ให้ traffic ถูกส่งเข้า pod ที่ยังไม่พร้อม (เช่นกำลัง warm-up connection pool กับ database) ส่วน liveness probe กันไม่ให้ pod ที่ deadlock หรือค้างอยู่เนิ่นนานโดยไม่ถูก restart พูดง่ายๆ คือ ทุกครั้งที่ deploy เวอร์ชันใหม่ ควรเช็กในขั้นตอน validation ว่า pod ผ่านสถานะ ready จริง ไม่ใช่แค่ status เป็น Running เฉยๆ เพราะ Running กับ Ready เป็นคนละสถานะกัน pod ที่ค้าง readiness probe จะไม่ถูกส่ง traffic เข้าเลย แม้จะขึ้น Running แล้วก็ตาม

Conformance Testing: สอบว่าคลัสเตอร์ "เป็น Kubernetes จริง" ไหม

อีกระดับหนึ่งที่มักถูกมองข้ามคือ conformance testing — ตรวจว่าคลัสเตอร์ (โดยเฉพาะคลัสเตอร์ที่ตั้งเองหรือใช้ distro ที่ไม่ใช่ managed service ของ cloud provider ใหญ่) ยังทำงานตรงตามมาตรฐาน Kubernetes API จริงหรือเปล่า เครื่องมือที่รู้จักกันในวงการนี้คือ sonobuoy ซึ่งรัน e2e conformance test suite อย่างเป็นทางการเข้ากับคลัสเตอร์ บทนี้จะไม่ลงลึกวิธีใช้ sonobuoy แบบเต็มรูปแบบ แค่อยากให้รู้จักไว้ว่ามีเครื่องมือนี้อยู่ — ถ้าทีมดูแล self-managed cluster หรือ custom distro เอง ควรกันเวลาไปศึกษาการรัน conformance test แยกต่างหากอีกที ก่อนจะประกาศว่าคลัสเตอร์นี้ "เป็น Kubernetes มาตรฐาน" จริง

เครื่องมือตอบคำถามอะไรรันตอนไหน
k6ระบบรับ traffic/load จริงไหวไหมก่อนขึ้น production, ก่อน event ที่คาด traffic สูง
Readiness/Liveness probepod ตัวนี้พร้อมรับ traffic ตอนนี้ไหม / ยัง alive อยู่ไหมทุกครั้งที่ deploy อัตโนมัติ ตลอดเวลาที่ pod รันอยู่
sonobuoyคลัสเตอร์นี้ตรงตามมาตรฐาน Kubernetes API จริงไหมตอนตั้งคลัสเตอร์ใหม่, หลังอัปเกรดเวอร์ชันใหญ่, ตรวจ compliance

💡 ทั้งสามระดับ ไม่มีตัวไหนแทนกันได้

ผ่าน load test ไม่ได้แปลว่า probe ตั้งค่าดีแล้ว และคลัสเตอร์ผ่าน conformance test ก็ไม่ได้แปลว่าแอปจะรับ load ไหว — สามอย่างนี้ตรวจคนละชั้น ต้องทำครบทั้งสามอย่างเป็นส่วนหนึ่งของ checklist ก่อนขึ้น production ไม่ใช่เลือกทำแค่อย่างเดียวแล้วมั่นใจว่าพร้อมแล้ว

ตอนนี้เรามีทั้ง backup ที่ซ้อม restore แล้วจริง มี monitoring ที่คอยเตือนก่อนพัง และมีชุดทดสอบที่ยืนยันว่าคลัสเตอร์รับ load ไหวก่อนปล่อยจริง — สามบทนี้รวมกันคือสิ่งที่แยก "คลัสเตอร์ที่ใช้งานได้" ออกจาก "คลัสเตอร์ที่พร้อม production จริง" ไว้ให้เห็นชัดๆ

Hardening ตาม best practice: RBAC, Pod Security, NetworkPolicy ต่างกันตรงไหน แล้วต้องใช้ครบทุกตัวจริงไหม?

บทนี้จะปิดรูโหว่ "ข้างใน" cluster ให้ครบ 3 ชั้น — ใครมีสิทธิ์ทำอะไร (RBAC), pod ทำอะไรกับเครื่องได้บ้าง (Pod Security Standards), และใครคุยกับใครได้ (NetworkPolicy) — ก่อนจะปิดท้ายด้วยเครื่องมือ verify ว่า hardening ที่ทำไปนั้น "ได้ผลจริง" หรือแค่รู้สึกว่าปลอดภัย

หลังจากบทที่แล้วสมชายเทส cluster จน chaos test ผ่านหมด ก็เริ่มมั่นใจว่าระบบพร้อม production เต็มที่ — จนกระทั่งฝ่าย security ขอเข้ามา audit ก่อน launch จริง แล้วส่งรายงานกลับมา...

😅
สมชายเห้ย นาย ก. รายงาน audit เขียนว่า ServiceAccount default ใน namespace production ผูก ClusterRoleBinding เป็น cluster-admin ไว้... นี่มันแปลว่าอะไรเหรอ
🧑‍🏫
นาย ก.แปลว่า pod ไหนก็ตามที่รันด้วย default ServiceAccount ใน namespace นั้น ถ้ามีคน exploit เข้าไปได้แม้แต่ตัวเดียว มันสามารถยิง kubectl API สั่ง delete/create อะไรก็ได้ทั้งคลัสเตอร์เลยนะ ไม่ใช่แค่ namespace เดียว

นี่คือปัญหาคลาสสิกของ cluster ที่รันมานานโดยไม่เคยรีวิว permission — ทุกอย่าง "รันได้" ก็เลยไม่มีใครไปยุ่งกับ RBAC จนกว่าจะมีคนมาชี้ให้เห็นว่ามันเปิดกว้างขนาดไหน ทีนี้ปัญหาแบบนี้ไม่ได้แก้ด้วยเครื่องมือตัวเดียว เพราะแต่ละเครื่องมือ "ปิดรู" คนละมิติกัน

กล่าวสั้น ๆ คือ: hardening ของ K8s ที่ดีต้องเป็น defense-in-depth หลายชั้นซ้อนกัน — RBAC ควบคุมว่า "ใคร/อะไร" เรียก API ได้บ้าง, Pod Security Standards ควบคุมว่า pod ที่รันแล้วทำอะไรกับ host ได้บ้าง, NetworkPolicy ควบคุมว่า pod คุยกับ pod ไหนได้บ้าง แล้วปิดท้ายด้วย kube-bench/Trivy เพื่อ verify ว่าที่ทำมาทั้งหมดตรง benchmark จริงไหม ไม่ใช่แค่ "รู้สึกว่าปลอดภัย"

1) RBAC แบบ least-privilege: เลิกให้ใครก็ได้เป็น cluster-admin

โจทย์ของ RBAC คือให้แต่ละ user/ServiceAccount มีสิทธิ์ "เท่าที่จำเป็น" เท่านั้น ตัวอย่างด้านล่างเป็น Role ที่จำกัดสิทธิ์แค่ "อ่าน pod" ใน namespace เดียว แล้วผูกเข้ากับ user ผ่าน RoleBinding — ไม่ใช่ ClusterRole/ClusterRoleBinding ที่จะกินสิทธิ์ทั้งคลัสเตอร์

rbac/pod-reader.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: production
subjects:
- kind: User
  name: jane
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

อธิบายจาก code

  1. Role — ตั้ง namespace ไว้ที่ production ชัดเจน สิทธิ์นี้จะมีผลแค่ namespace เดียว ไม่กระทบ namespace อื่น
  2. rules — apiGroups: [""] คือ core API group, resources: ["pods"] จำกัดแค่ resource pod, verbs อนุญาตแค่ get/watch/list คือ "อ่านอย่างเดียว" ไม่มี create/delete/update
  3. RoleBinding — ผูก subjects (user jane) เข้ากับ roleRef (Role ชื่อ pod-reader) ทั้งคู่ต้องอยู่ใน namespace เดียวกัน
  4. ถ้าจะให้สิทธิ์ทั้งคลัสเตอร์จริง ๆ ต้องเปลี่ยนเป็น ClusterRole + ClusterRoleBinding แทน — ซึ่งคือสิ่งที่ ServiceAccount default ของสมชายโดนผูกไว้แบบผิด ๆ นั่นเอง

2) Pod Security Standards: pod รันแล้วทำอะไรกับ host ได้บ้าง

RBAC คุมแค่ "ใครเรียก API ได้" แต่ไม่ได้คุมว่า pod ที่รันอยู่แล้วจะ mount hostPath, รันเป็น root, หรือขอ privileged container ได้ไหม — นี่คือหน้าที่ของ Pod Security Standards (PSS) ซึ่งมี 3 ระดับ

🔓 Privileged

ไม่จำกัดอะไรเลย เปิดกว้างเต็มที่ ใช้เฉพาะ workload ระบบที่ต้องคุม host จริง ๆ (เช่น CNI plugin)

🟡 Baseline

จำกัดแบบขั้นต่ำ กันช่องโหว่ privilege escalation ที่รู้จักกันแล้ว เหมาะเป็นค่าเริ่มต้นของ namespace ทั่วไป

🔒 Restricted

เข้มสุด ตาม pod hardening best practice ปัจจุบัน — ห้ามรันเป็น root, ห้าม privilege escalation, จำกัด volume type ที่ใช้ได้

วิธีบังคับใช้ทำผ่าน label บน namespace ตรง ๆ ไม่ต้องติดตั้ง admission controller เพิ่มเอง (K8s 1.23+ มี Pod Security admission ในตัว) — คีย์ label คือ pod-security.kubernetes.io/<mode> โดย mode มี 3 แบบคือ enforce (บล็อกจริง), audit (log ไว้เฉย ๆ), warn (เตือนตอน apply) แต่ละ mode มี suffix -version ให้ pin เวอร์ชันได้ด้วย

namespace-production.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: v1.31
    pod-security.kubernetes.io/warn: restricted

อธิบายจาก code

  1. pod-security.kubernetes.io/enforce: restricted — บล็อก pod ที่ไม่ผ่านระดับ restricted ตั้งแต่ตอน apply เลย ไม่ใช่แค่เตือน
  2. enforce-version: v1.31 — pin เวอร์ชันของ policy ให้ตรงกับตอนที่ทดสอบไว้ ป้องกัน behavior เปลี่ยนแบบไม่รู้ตัวตอน K8s อัปเกรด
  3. warn: restricted — ใช้คู่กับ enforce เพื่อให้ผู้ที่ apply เห็น warning ชัดเจนตอนที่ manifest มีอะไรขัดกับ restricted แม้จะยัง apply ผ่าน (เผื่อ enforce ตั้งไว้ต่ำกว่า)

ถ้า namespace มีอยู่แล้วไม่อยากรื้อ manifest ทั้งไฟล์ ก็ label เพิ่มทีหลังได้ตรง ๆ ผ่าน kubectl:

terminal
kubectl label --overwrite ns production \
  pod-security.kubernetes.io/enforce=restricted \
  pod-security.kubernetes.io/enforce-version=v1.31

อธิบายจาก code

  1. --overwrite — จำเป็นเพราะ namespace อาจมี label ชื่อนี้อยู่แล้ว (เช่นตั้งเป็น baseline ไว้ก่อน) ถ้าไม่ใส่ flag นี้ kubectl จะปฏิเสธการเขียนทับ
  2. คำสั่งเดียวใส่ได้หลาย label พร้อมกัน — สะดวกเวลาต้องตั้ง enforce + enforce-version พร้อมกันในทีเดียว

⚠️ เวอร์ชันต้อง pin ให้ตรงคลัสเตอร์จริง

v1.31 ในตัวอย่างเป็นแค่ตัวอย่าง ต้องเช็คเวอร์ชัน minor ของ K8s cluster ตัวเองก่อนใส่ (kubectl version) เพราะ policy ของแต่ละเวอร์ชันมีรายละเอียดต่างกัน ใส่เวอร์ชันผิดอาจทำให้ enforce ทำงานไม่ตรงกับที่ทดสอบไว้

3) NetworkPolicy: default-deny-ingress ก่อนเปิดเฉพาะที่จำเป็น

ทีนี้ต่อให้ RBAC กับ PSS แน่นแค่ไหน ถ้า pod ทุกตัวใน cluster ยังคุยกันได้อย่างอิสระ (ค่า default ของ K8s คือ "ไม่บล็อกอะไรเลย") แปลว่า pod ที่โดน exploit ตัวเดียวสามารถไล่สแกนหา pod อื่นในเครือข่ายเดียวกันได้หมด แนวทางมาตรฐานคือเริ่มจาก deny-all แล้วค่อยเปิดเฉพาะ path ที่จำเป็น

networkpolicy/default-deny-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress

อธิบายจาก code

  1. podSelector: {} — selector ว่าง หมายถึง "pod ทุกตัวใน namespace นี้" ไม่ใช่ "ไม่เลือกใครเลย"
  2. policyTypes: [Ingress] — ประกาศว่า policy นี้ควบคุมขาเข้า (ingress traffic) เท่านั้น
  3. เพราะไม่มี ingress: rule ระบุไว้เลยข้างล่าง ผลคือ "ไม่มี traffic ขาเข้าใดเข้าได้เลย" — นี่คือ default-deny ที่แท้จริง
  4. หลังจากนี้ต้องสร้าง NetworkPolicy เพิ่มเติมเพื่อ "อนุญาต" เฉพาะ path ที่จำเป็น เช่น ให้ Ingress controller คุยกับ pod แอปได้ ให้ pod แอปคุยกับ database ได้ เป็นต้น

💡 กติกาเรื่อง OR / AND ของ NetworkPolicy

ตอนเขียน rule เพิ่มเติมให้จำไว้ว่า entry หลายอันใน from/to จะถูก OR กัน (เข้าเงื่อนไขอันใดอันหนึ่งก็พอ) แต่ field ต่าง ๆ ภายใน entry เดียวกันจะถูก AND กัน (ต้องตรงทุก field ใน entry นั้น) — เขียนผิดจุดนี้บ่อยมากจนเปิดกว้างเกินตั้งใจ

4) kube-bench: เช็คว่า cluster เราผ่าน CIS Benchmark จริงไหม

ตั้งค่าทั้งหมดข้างบนมาด้วยมือ แล้วจะรู้ได้ไงว่าตั้งครบตาม standard ที่ยอมรับกันในวงการ? kube-bench คือเครื่องมือที่เช็คว่า Kubernetes ถูก deploy อย่างปลอดภัยตามเกณฑ์ของ CIS Kubernetes Benchmark วิธีรันที่สะดวกสุดในคลัสเตอร์คือรันเป็น Job

terminal
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml
kubectl get pods
kubectl logs job/kube-bench

อธิบายจาก code

  1. kubectl apply -f .../job.yaml — ดึง manifest Job ตัวจริงจาก repo ทางการมาสร้าง Job ที่รัน kube-bench เป็น pod ในคลัสเตอร์ตรง ๆ
  2. kubectl get pods — เช็คว่า pod ของ Job รันจบ (Completed) หรือยัง
  3. kubectl logs job/kube-bench — ดึง log ผลลัพธ์ ซึ่งจะไล่ลิสต์ทีละ check ตามหมวดของ CIS benchmark พร้อมสถานะ PASS / FAIL / WARN / INFO ต่อรายการ ให้ไล่ดู FAIL ก่อนเป็นอันดับแรก

ถ้าไม่อยากรันเป็น Job ในคลัสเตอร์ ก็มีทางเลือกรันแบบ container ตรงบน node ผ่าน Docker ได้เหมือนกัน โดยต้อง mount /etc กับ /var แบบ read-only เข้าไปให้ kube-bench อ่าน config ของ node จริง:

terminal
docker run --pid=host -v /etc:/etc:ro -v /var:/var:ro -t docker.io/aquasec/kube-bench:latest --version 1.18

อธิบายจาก code

  1. --pid=host — ให้ container มองเห็น process ของ host เพื่อเช็ค flag ที่ kube-apiserver/kubelet รันอยู่จริง
  2. -v /etc:/etc:ro -v /var:/var:ro — mount config file ของ node เข้ามาแบบอ่านอย่างเดียว
  3. --version 1.18 — ระบุเวอร์ชันอ้างอิงให้ kube-bench ใช้เลือกชุด check ที่เหมาะสม (แยกจาก flag --benchmark ที่ใช้ระบุชื่อ CIS benchmark ตรง ๆ) ไม่ใช่เลขเวอร์ชันของตัวโปรแกรม kube-bench เอง — ให้ปรับเลขนี้ให้ตรงกับคลัสเตอร์จริงเสมอ

💡 benchmark version ต้องเลือกให้ตรง K8s version

CIS benchmark ออกเป็นเวอร์ชันตาม K8s minor version (flag --benchmark/--version ต่างกันไปตามคู่มือ) — รันด้วย benchmark คนละเวอร์ชันกับคลัสเตอร์จริงจะได้ผล FAIL/PASS ที่เข้าใจผิดได้ ให้เช็คเวอร์ชันคลัสเตอร์ก่อนเลือก benchmark ที่จะรันเสมอ

5) Trivy: scan image ก่อน deploy และ scan ทั้ง cluster

kube-bench เช็คการตั้งค่าคลัสเตอร์ แต่ไม่ได้เช็คว่า image ที่กำลังจะ deploy มี CVE เก่าซุกอยู่ไหม นี่คือหน้าที่ของ Trivy ซึ่งใช้ scan ได้ทั้งระดับ image เดี่ยว ๆ และทั้งคลัสเตอร์รวด

terminal
trivy image myregistry/nextjs-app:1.0.0

อธิบายจาก code

  1. trivy image — scan image ที่ระบุ (ในที่นี้คือ image ของ Next.js app จากบทก่อน ๆ) หา CVE ในทุก layer ทั้ง OS package และ library ของแอป ควรรันเป็น step ใน CI ก่อน push image ขึ้น registry
terminal
trivy k8s --report summary cluster

อธิบายจาก code

  1. trivy k8s cluster — สแกน resource ทั้งคลัสเตอร์ (ไม่ใช่แค่ image เดียว) รวมถึง config ของ Role/RoleBinding, Pod spec ที่รันอยู่จริง
  2. --report summary — สรุปผลแบบภาพรวมต่อ namespace/resource แทนที่จะพ่น finding รายละเอียดยาวทั้งหมด เหมาะกับดูภาพรวมก่อนไล่ลึก

⚠️ trivy k8s ยังเป็น EXPERIMENTAL

official docs ของ Trivy ระบุชัดว่าฟีเจอร์ trivy k8s ยังอยู่สถานะ EXPERIMENTAL "อาจเปลี่ยนแปลงได้โดยไม่รักษาความเข้ากันได้แบบย้อนหลัง" (might change without preserving backwards compatibility) — เหมาะใช้เป็นเครื่องมือช่วย audit เพิ่มเติม แต่อย่าผูก pipeline สำคัญไว้กับ output format ของคำสั่งนี้แบบตายตัว เพราะ flag/format อาจเปลี่ยนได้ในเวอร์ชันถัดไป

ตอนนี้ cluster ของสมชายปิดรูข้างในครบแล้ว — ใครเรียก API ได้แค่เท่าที่จำเป็น (RBAC), pod ทำอะไรกับ host ได้จำกัด (PSS), pod คุยกันได้แค่ path ที่อนุญาต (NetworkPolicy) แล้วก็มีเครื่องมือ verify (kube-bench/Trivy) คอยเช็คซ้ำ แต่ทั้งหมดนี้เป็นการป้องกัน "ข้างใน" คลัสเตอร์ล้วน ๆ — traffic จากโลกภายนอกที่วิ่งเข้ามาทาง HTTP ยังไม่มีใครกรองเลยสักตัว บทต่อไปจะไปปิด layer สุดท้ายนี้กัน

🛡️ บทที่ 13

Proxy + WAF: Ingress-NGINX, cert-manager, ModSecurity ต่างหน้าที่กันยังไง แล้วประกอบร่างเป็นปราการด่านสุดท้ายได้ยังไง

บทนี้คือ layer ป้องกันสุดท้ายก่อน traffic จากอินเทอร์เน็ตจะแตะ pod จริง — ติดตั้ง Ingress-NGINX เป็นทางเข้าเดียว, ใช้ cert-manager ออก TLS cert อัตโนมัติ, เปิด ModSecurity + OWASP Core Rule Set เป็น WAF กรอง request อันตรายระดับ L7 และปิดท้ายด้วย rate limiting

หลังบทที่แล้ว cluster ของสมชายแน่นขึ้นมากทั้ง RBAC, Pod Security, NetworkPolicy จนตัวเองก็เริ่มมั่นใจ แต่นาย ข. ที่เพิ่งเข้าทีมมาถามคำถามที่ฟังดูไร้เดียงสาแต่จริง ๆ แหลมมาก

นาย ก.เพราะ RBAC/PSS/NetworkPolicy ทั้งหมดที่ทำมาคุมแค่ "ข้างใน" คลัสเตอร์ — ไม่มีตัวไหนอ่านเนื้อ HTTP request เลยสักตัว มันไม่รู้หรอกว่า body ของ request มี payload SQLi ซ่อนอยู่ ต้องมี layer ที่อ่าน L7 จริง ๆ ถึงจะกรองแบบนี้ได้

คำตอบของนาย ก. คือกุญแจของบทนี้ — ทุกอย่างที่ทำมาในบทก่อนเป็นการป้องกันระดับ infrastructure/network แต่ layer สุดท้ายที่ต้องมีคือจุดที่ request จริงวิ่งผ่าน ซึ่งต้องทำ 4 อย่างพร้อมกัน คือมี "ประตูทางเข้าเดียว" ที่เข้ารหัสด้วย TLS, มี "ยาม" ที่อ่านเนื้อ request กรอง payload อันตราย และมี "ตัวจำกัดจำนวนคนเข้าคิว" กัน brute-force

กล่าวสั้น ๆ คือ: Ingress-NGINX เป็นประตูทางเข้าเดียวของ traffic ทั้งหมด, cert-manager ทำหน้าที่ออก/ต่ออายุ TLS cert ให้อัตโนมัติไม่ต้องขอมือ, ModSecurity + OWASP CRS คือ WAF ที่อ่านเนื้อ request กรอง payload อันตรายระดับ L7, ส่วน rate-limit annotation ช่วยกัน brute-force/DDoS เบื้องต้น — ทั้งหมดนี้ประกอบกันเป็น ปราการด่านสุดท้ายก่อนถึง pod จริง

Ingress-NGINXreverse proxy + entry point เดียวของ clusterไม่ต้อง expose แต่ละ Service ออกเน็ตตรง ๆ

1) ติดตั้ง Ingress-NGINX ด้วย Helm

เริ่มจากติดตั้งตัว controller เองก่อนผ่าน Helm chart ทางการ

terminal
helm upgrade --install ingress-nginx ingress-nginx \
  --repo https://kubernetes.github.io/ingress-nginx \
  --namespace ingress-nginx --create-namespace

อธิบายจาก code

  1. helm upgrade --install — ใช้แทน helm install ตรง ๆ เพราะรันซ้ำได้ปลอดภัย ถ้ายังไม่เคยติดตั้งจะ install ให้ ถ้าติดตั้งแล้วจะ upgrade แทน
  2. --repo https://kubernetes.github.io/ingress-nginx — ดึง chart จาก Helm repo ทางการของโปรเจกต์ตรง ๆ ไม่ต้อง helm repo add ล่วงหน้า
  3. --namespace ingress-nginx --create-namespace — แยก namespace เฉพาะให้ controller ไม่ปนกับ workload อื่น พร้อมสร้าง namespace ให้อัตโนมัติถ้ายังไม่มี

จากนั้นสร้าง Ingress resource ตัวแรกให้ชี้เข้า Service ของ Next.js app ที่ deploy ไว้ตั้งแต่บทก่อน ๆ

ingress/nextjs-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: nextjs-ingress
spec:
  ingressClassName: nginx
  rules:
  - host: app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: nextjs-app
            port:
              number: 80

อธิบายจาก code

  1. ingressClassName: nginx — บอกว่า Ingress นี้ให้ Ingress-NGINX controller เป็นคนดูแล (เผื่อคลัสเตอร์มีหลาย controller ติดตั้งพร้อมกัน)
  2. rules[].host — domain ที่ Ingress นี้รับ traffic (app.example.com) — request ที่ Host header ไม่ตรงจะไม่ถูกส่งมาที่ rule นี้
  3. paths[].path: / + pathType: Prefix — จับทุก path ที่ขึ้นต้นด้วย / คือทั้งเว็บ
  4. backend.service — ส่งต่อไปที่ Service ชื่อ nextjs-app พอร์ต 80 ซึ่งเป็น Service ที่ตั้งไว้ตั้งแต่บท deploy Next.js

2) cert-manager + Let's Encrypt: TLS อัตโนมัติ ไม่ต้องขอ cert มือ

มี Ingress แล้วแต่ยังเป็น HTTP เปลือย ๆ ทีนี้ติดตั้ง cert-manager เพื่อขอ/ต่ออายุ cert จาก Let's Encrypt ให้อัตโนมัติ

terminal
helm repo add jetstack https://charts.jetstack.io --force-update
helm install cert-manager jetstack/cert-manager \
  --namespace cert-manager --create-namespace \
  --set crds.enabled=true

อธิบายจาก code

  1. helm repo add jetstack ... --force-update — เพิ่ม Helm repo ทางการของ cert-manager พร้อม force update ให้ได้ chart index ล่าสุดเสมอ
  2. --namespace cert-manager --create-namespace — แยก namespace เฉพาะเหมือน ingress-nginx
  3. --set crds.enabled=true — สั่งให้ chart ติดตั้ง CustomResourceDefinition (เช่น Certificate, ClusterIssuer) มาพร้อมกันในทีเดียว ไม่ต้อง apply แยก

💡 flag เปลี่ยนชื่อ อย่าใช้ของเก่า

docs รุ่นเก่า/บทความเก่าจำนวนมากยังใช้ --set installCRDs=true ซึ่งเป็น flag เก่าของ cert-manager — เอกสารทางการปัจจุบันเปลี่ยนมาใช้ --set crds.enabled=true แล้ว ถ้า copy คำสั่งจากบทความเก่ามาใช้ตรง ๆ อาจ error หรือ CRD ไม่ถูกติดตั้งตามที่คาดไว้

ติดตั้งเสร็จแล้วต้องสร้าง ClusterIssuer เพื่อบอกว่าจะขอ cert จากที่ไหน ด้วยวิธี challenge แบบไหน

cert-manager/clusterissuer-letsencrypt.yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-prod
spec:
  acme:
    server: https://acme-v02.api.letsencrypt.org/directory
    email: admin@example.com
    privateKeySecretRef:
      name: letsencrypt-prod
    solvers:
      - http01:
          ingress:
            ingressClassName: nginx

อธิบายจาก code

  1. server — ชี้ไปที่ ACME endpoint ตัวจริง (production) ของ Let's Encrypt — ถ้าเทสอยู่ควรใช้ staging endpoint แยกต่างหากเพื่อไม่ให้ชน rate limit ของ production
  2. email — อีเมลรับแจ้งเตือนก่อน cert หมดอายุ/มีปัญหาการต่ออายุ
  3. privateKeySecretRef — ชื่อ Secret ที่ cert-manager จะเก็บ private key ของ account ACME ไว้ใช้ครั้งถัดไป
  4. solvers[].http01.ingress.ingressClassName: nginx — เลือกวิธี validate domain แบบ HTTP-01 โดยฝาก challenge ไว้ผ่าน Ingress-NGINX controller ที่ติดตั้งไว้ในขั้นก่อนหน้า

สุดท้ายกลับไปเติม annotation กับ block tls ที่ Ingress เดิมเพื่อเปิด auto-TLS

ingress/nextjs-ingress.yaml (เพิ่ม TLS)
metadata:
  annotations:
    cert-manager.io/cluster-issuer: "letsencrypt-prod"
spec:
  tls:
  - hosts:
    - app.example.com
    secretName: nextjs-app-tls

อธิบายจาก code

  1. cert-manager.io/cluster-issuer: "letsencrypt-prod" — บอก cert-manager ว่า Ingress นี้ต้องการ cert จาก ClusterIssuer ที่สร้างไว้ก่อนหน้า
  2. tls[].hosts — ต้องตรงกับ host ใน rules เดิม ไม่งั้น cert จะออกมาไม่ตรง domain ที่ใช้จริง
  3. secretName: nextjs-app-tls — cert-manager จะสร้าง Secret ชื่อนี้ให้อัตโนมัติ เก็บทั้ง cert และ private key แล้ว Ingress-NGINX จะดึง Secret นี้มาทำ TLS termination เอง

3) ModSecurity + OWASP Core Rule Set: WAF ระดับ L7

ตอนนี้มี TLS แล้ว แต่ยังไม่มีใคร "อ่านเนื้อ" request เลย — เปิด ModSecurity พร้อม OWASP Core Rule Set (CRS) ผ่าน annotation ของ Ingress-NGINX ได้ตรง ๆ ทีละ Ingress

ingress/nextjs-ingress.yaml (เพิ่ม WAF)
metadata:
  annotations:
    nginx.ingress.kubernetes.io/enable-modsecurity: "true"
    nginx.ingress.kubernetes.io/enable-owasp-core-rules: "true"
    nginx.ingress.kubernetes.io/modsecurity-snippet: |
      SecRuleEngine On
      SecDebugLog /tmp/modsec_debug.log

อธิบายจาก code

  1. enable-modsecurity: "true" — เปิด ModSecurity module ให้ Ingress นี้ (ค่า default ของทั้ง controller คือปิดไว้)
  2. enable-owasp-core-rules: "true" — โหลดชุด rule OWASP CRS เข้ามาใช้กรอง pattern อันตรายที่รู้จักกัน (SQLi, XSS ฯลฯ)
  3. modsecurity-snippet — เพิ่ม config ดิบของ ModSecurity ต่อ Ingress ในที่นี้สั่ง SecRuleEngine On เพื่อให้ ModSecurity "บล็อกจริง" ไม่ใช่แค่ log

⚠️ ชื่อ annotation กับชื่อคีย์ใน ConfigMap ไม่เหมือนกัน

จุดที่งงกันบ่อยมาก — ตอนเปิดแบบ per-Ingress annotation ใช้ชื่อ enable-owasp-core-rules แต่ถ้าจะเปิดแบบ global ผ่าน ConfigMap ของ controller (enable-modsecurity: "false" และ enable-owasp-modsecurity-crs: "false" เป็นค่า default ปิดไว้ทั้งคู่) ต้องใช้ชื่อคีย์ enable-owasp-modsecurity-crs แทน คนละชื่อกันแม้จะทำหน้าที่เดียวกัน ใส่ผิดชื่อจะไม่ error แต่ CRS จะไม่ถูกโหลดเงียบ ๆ โดยไม่มีใครรู้ตัว

⚠️ ค่า default คือ Detection-Only ไม่ใช่ Block

พอเปิด ModSecurity แล้ว โหมดเริ่มต้นคือ Detection-Only คือ log ว่าเจอ pattern น่าสงสัยไว้เฉย ๆ แต่ยังปล่อย request ผ่านไปตามปกติ — ถ้าต้องการให้บล็อกจริงต้องสั่ง SecRuleEngine On เพิ่มเองแบบในตัวอย่างข้างบน แนะนำให้รันโหมด Detection-Only สักพักก่อนเพื่อดู false positive ก่อนสลับไปโหมดบล็อกจริงบน production

4) Rate limiting: กัน brute-force / DDoS เบื้องต้นระดับ Ingress

WAF กรอง payload อันตรายได้ แต่ไม่ได้กันคนยิง request รัว ๆ เพื่อ brute-force password หรือถล่ม endpoint เดียวซ้ำ ๆ — annotation ของ Ingress-NGINX มีให้ตั้ง rate limit ต่อ IP ได้ตรง ๆ

ingress/nextjs-ingress.yaml (เพิ่ม rate limit)
metadata:
  annotations:
    nginx.ingress.kubernetes.io/limit-rps: "20"
    nginx.ingress.kubernetes.io/limit-burst-multiplier: "5"
    nginx.ingress.kubernetes.io/limit-connections: "10"

อธิบายจาก code

  1. limit-rps: "20" — จำกัด request ต่อวินาทีต่อ IP หนึ่งไม่เกิน 20 ครั้ง
  2. limit-burst-multiplier: "5" — อนุญาตให้ burst ได้สูงสุด 5 เท่าของ limit-rps ชั่วครู่ (ค่า default ของ controller ก็คือ 5 เท่าอยู่แล้ว ใส่ไว้ตรง ๆ ให้ชัดเจน) กัน false positive ตอน user โหลดหน้าเว็บที่มีหลาย asset พร้อมกัน
  3. limit-connections: "10" — จำกัดจำนวน connection พร้อมกันต่อ IP ต่อ replica ไม่เกิน 10 connection

รวมทั้ง 4 ชั้นเข้าด้วยกัน เส้นทางของ request หนึ่งตัวตอนนี้จะหน้าตาแบบนี้ — จากลูกค้าวิ่งผ่าน WAF/Ingress-NGINX ก่อนถึง Service แล้วค่อยเข้า Pod จริง ไม่มี path ไหนที่ข้ามชั้นกรองไปได้เลย

ตอนนี้ cluster ของสมชายมีครบทั้งเกราะชั้นใน (RBAC, Pod Security, NetworkPolicy) และเกราะชั้นนอก (Ingress-NGINX, cert-manager, ModSecurity, rate limit) แล้ว บทต่อไปจะเอาทุกชิ้นส่วนตั้งแต่บทที่ 2 มาจนถึงบทนี้ มาประกอบร่างเป็นระบบ production เดียวจริง ๆ ให้เห็นภาพรวมทั้งหมดพร้อมกัน

13 บทที่ผ่านมาแต่ละบทแก้ปัญหาคนละมิติ บทนี้เอาทุกชิ้นมาต่อกันเป็นภาพเดียว ให้เห็นว่าคลัสเตอร์ที่ "production-grade" จริงๆ หน้าตาเป็นยังไงเมื่อมองจากบนลงล่าง

🧑‍🏫
นาย ก.ได้เลยครับ ลองมองเป็นสองมุม — มุมแรกคือ "โครงสร้างเครื่อง" ว่ามี node อะไรบ้าง อีกมุมคือ "เส้นทางของ request" ว่าวิ่งผ่านอะไรบ้างกว่าจะถึงแอปจริง

กล่าวสั้นๆ คือ production cluster ที่สมบูรณ์ไม่ใช่แค่ "รันแอปได้" แต่ต้องตอบได้ครบ 4 คำถาม — ทนต่อความล้มเหลวได้ไหม (topology), backup กู้คืนได้จริงไหม (Velero + etcd), มองเห็นปัญหาก่อนลูกค้ารู้ไหม (monitoring), และป้องกันการโจมตีได้ไหม (hardening + WAF) — เหมือนที่พูดไว้ตั้งแต่บทที่ 1

มุมที่ 1 — โครงสร้างเครื่องทั้งคลัสเตอร์

มุมที่ 2 — เส้นทางของ request หนึ่งตัว ตั้งแต่ลูกค้ายิงเข้ามาจนถึง Pod

สองมุมนี้คือ "โครงสร้างนิ่ง" กับ "การไหลของ traffic" ส่วนที่มองไม่เห็นในไดอะแกรมแต่ทำงานอยู่ตลอดเวลาคือสามระบบเสริมที่ครอบทั้งคลัสเตอร์อยู่ — Velero + etcd snapshot (บทที่ 9) คอย backup ทุกวันตี 3, kube-prometheus-stack + Loki (บทที่ 10) คอยเฝ้ามองและแจ้งเตือน, และ k6 + probe (บทที่ 11) คอยยืนยันก่อน deploy ทุกครั้งว่าของจริงรับ load ไหว

kubectl exec -it <pod-name> -n <namespace> -- sh

Deploy และ Rollback

terminal
# อัปเดต image เวอร์ชันใหม่แบบ rolling update
kubectl set image deployment/nextjs-app nextjs-app=myregistry/nextjs-app:1.1.0

# ดูสถานะ rollout ว่าเสร็จหรือยัง
kubectl rollout status deployment/nextjs-app

# ย้อนกลับเวอร์ชันก่อนหน้าทันทีถ้า rollout ใหม่มีปัญหา
kubectl rollout undo deployment/nextjs-app

# ดูประวัติ revision ทั้งหมด
kubectl rollout history deployment/nextjs-app

Scale และ Resource

terminal
# scale replicas ด้วยมือ (เผื่อ HPA ยังไม่ทัน)
kubectl scale deployment/nextjs-app --replicas=5

# ดูการใช้ CPU/memory จริงของแต่ละ pod (ต้องมี metrics-server)
kubectl top pods -n production

# ดูสถานะ HPA แบบ real-time
kubectl get hpa nextjs-app -w

🎬 บทที่ 16

สรุปบทเรียน — จากคลัสเตอร์ที่ล่มตอนตีสาม ถึงระบบที่ทนต่อความล้มเหลวได้จริง

ย้อนกลับไปบทที่ 1 สมชายมีแค่ VM เดียวที่บังเอิญไม่ล่มมา 8 เดือน มาถึงตรงนี้เขามีคลัสเตอร์ที่ผ่าน 4 มิติของ production-grade ครบแล้ว บทนี้สรุปบทเรียนสำคัญที่สุดจากแต่ละส่วน

🏗️ Architecture

แยก control plane กับ node component ออกจากกันด้วยเหตุผล ไม่ใช่แค่ตามสอนกันมา — แต่ละตัวมี failure mode ต่างกัน

🧭 Topology

HA ผูกกับ etcd quorum โดยตรง — ต้องมี 3 เครื่องขึ้นไปเสมอ ไม่มีทางลัดที่ 2 เครื่อง

🖥️ VM Prep

containerd 1.x/2.x path ต่างกันเงียบๆ — เช็คเวอร์ชันก่อนแก้ config ทุกครั้ง

🚀 Bootstrap

repo เก่า packages.cloud.google.com ตายแล้ว ใช้ pkgs.k8s.io เท่านั้น

🌐 Networking

bare-metal ไม่มี cloud-controller-manager ให้พึ่ง — MetalLB คือตัวเติมช่องว่างนั้น

📦 Deploy Next.js

output: 'standalone' + probe + resources ครบ คือพื้นฐานที่ทุก production Deployment ต้องมี

🧰 Tooling

Helm สำหรับแจก chart, Kustomize สำหรับ manifest ของทีมเอง — ใช้ผสมกันได้

🛟 Backup/DR

backup ที่ไม่เคย restore ก็เหมือนไม่มี backup — ซ้อมกู้คืนอย่างน้อยไตรมาสละครั้ง

📡 Monitoring

คลัสเตอร์ต้อง "พูดได้" ก่อนลูกค้าจะเป็นคนบอกเราว่าระบบพัง

🧪 Testing

CI เขียวไม่เท่ากับพร้อม production — ต้องมี load test, probe validation, conformance test ครบ

🔒 Hardening

defense-in-depth หลายชั้น RBAC + PSS + NetworkPolicy — ปิดรูข้างในให้ครบก่อนมองข้างนอก

🛡️ Proxy + WAF

infrastructure hardening ทั้งหมดคุมไม่ถึงเนื้อ HTTP request — WAF คือ layer ที่อ่าน L7 จริง

🧑‍💻
สมชายย้อนกลับไปดูตอนตีสามคืนนั้น... ถ้าตอนนั้นผมมีทุกอย่างที่เพิ่งทำมา คงไม่ต้องตื่นมาแก้ปัญหาคนเดียวแบบนั้นเลยครับ
🧑‍🏫
นาย ก.นั่นแหละคือเป้าหมายจริงๆ ของ production-grade — ไม่ใช่ทำให้ระบบไม่มีวันพัง เพราะไม่มีระบบไหนไม่พังเลย แต่ทำให้ตอนที่มันพัง คุณมี backup กู้คืนได้ มี alert ที่บอกก่อนลูกค้าจะรู้ และมี node สำรองที่รับงานต่อได้ทันที — พังได้ แต่ล้มไม่ได้

กล่าวสั้นๆ คือ production-grade Kubernetes ไม่ใช่ปลายทางที่ทำครั้งเดียวจบ แต่เป็นชุดพฤติกรรมที่ต้องทำซ้ำสม่ำเสมอ — ซ้อม restore ทุกไตรมาส, รีวิว RBAC ทุกครั้งที่มีคนใหม่เข้าทีม, อัปเดตเวอร์ชัน CNI/MetalLB/Calico ตามรอบ, และอ่าน alert ที่ยิงเข้ามาจริงจัง ไม่ใช่แค่ตั้งไว้แล้วลืม
